# 0.6.7 тАФ Schema Precision Hints

**Status:** spec-locked, ready to implement
**Target:** 0.6.7 (between 0.6.5 Live Notifications and 0.7.0 npm prep)
**Owner:** agt-claude-dev
**Origin:** fup-2026-039
**Ship gate:** required before hosted brain goes live

## Problem

MAADB has exactly one datetime precision: whatever the writer happened to type. The engine stores `2026-04-16` and `2026-04-16T17:20:00.000Z` side-by-side in the same `date` field without comment. That's fine for storage (ISO8601 self-describes its own precision) but breaks three things:

1. **Future analytics.** Duration/heatmap/SLA features can't retroactively recover precision that was never captured. A journal entry that stored only `2026-04-15` as its timestamp has lost information permanently.
2. **LLM write quality.** Agents default to day-precision even when the schema's intent was "log the exact moment." Nothing signals otherwise.
3. **Consumer display flexibility.** Apps that want to show "Apr 15" for some fields and "Apr 15, 5:20 PM" for others today have to hardcode that logic тАФ the schema has nothing to guide them.

## Solution shape

Two orthogonal knobs on `date` field definitions. **Do not conflate them.**

### Store precision тАФ engine-enforced

Declared minimum precision on writes. Engine either warns or rejects writes coarser than declared.

```yaml
fields:
  started_at:
    type: date
    store_precision: second      # engine enforces minimum
    on_coarser: warn             # default; 'error' for strict
```

Rule: **always store the richest precision available.** Storage is cheap; data not captured is gone forever.

### Display precision тАФ consumer-side hint

Optional schema-level suggestion for how UIs should render. Engine returns full value regardless; render-time truncation is the consumer's job.

```yaml
fields:
  decided_at:
    type: date
    store_precision: second      # still capture rich storage
    on_coarser: warn
    display_precision: day       # typical display is just the date
```

Consumers honor the hint or ignore it. Users override per-field in consumer-app settings (e.g. "show dates only" тАФ brain-app territory, not MAADB's concern).

## Precision enum

Seven levels, ordered finest тЖТ coarsest inverse:

```
millisecond > second > minute > hour > day > month > year
```

Canonical detection from an ISO8601 string тАФ honor **the literal string's precision**, not the parsed Date object. `2026-04-16T00:00:00Z` is second-precision (because seconds are written), not day-precision-padded-to-midnight.

| Literal string pattern | Detected precision |
|---|---|
| `2026` | year |
| `2026-04` | month |
| `2026-04-16` | day |
| `2026-04-16T17` | hour |
| `2026-04-16T17:20` | minute |
| `2026-04-16T17:20:00` (with or without `Z`) | second |
| `2026-04-16T17:20:00.000` (with or without `Z`) | millisecond |

Timezone suffix (`Z`, `+00:00`) does not change the precision classification.

## Schema DSL additions

Three new optional keys on `date` field definitions. All backward-compatible тАФ absence = current lenient behavior.

| Key | Type | Default | Semantics |
|---|---|---|---|
| `store_precision` | enum (7 levels) | unset | If set, enforce minimum precision on write. If unset, no enforcement. |
| `on_coarser` | `warn \| error` | `warn` | When `store_precision` is violated, warn logs + surfaces in `_meta.warnings[]`; error rejects with `VALIDATION_FAILED`. |
| `display_precision` | enum (7 levels) | unset | Passthrough hint for consumers. Engine never enforces. Must be coarser-or-equal to `store_precision` (validated at schema load). |

**Schema-load validation rule:** if both keys are present, `display_precision` must be тЙе `store_precision` (i.e. display no finer than storage). Inverted тЖТ reject the schema file at load.

## The backward-compat invariant (load-bearing)

**Precision enforcement fires at write-time only. Never on read. Never on reindex. Never on `maad_validate` as error тАФ only as informational report.**

This is the core rule. Violate it and every existing brain with historical day-precision data breaks the moment any schema adopts `store_precision`.

### Call-site audit (5 places `validateFrontmatter` fires today)

### Implementation mechanism тАФ mode parameter on validator

Extend `validateFrontmatter` signature with an optional `options` object:

```typescript
interface ValidationOptions {
  mode: 'write' | 'read' | 'index' | 'audit';
  changedFields?: Set<string>;  // update-path only
}

function validateFrontmatter(
  frontmatter: Record<string, unknown>,
  schema: SchemaDefinition,
  registry: Registry,
  filePath?: FilePath,
  options?: ValidationOptions,  // default { mode: 'read' } тАФ safe default
): ValidationResult;
```

**Precision enforcement gate:**

```typescript
if (options?.mode !== 'write') return;             // no enforcement off write path
if (!fieldDef.storePrecision) return;              // no enforcement unless declared
if (options.changedFields && !options.changedFields.has(fieldName)) return;  // update: skip unchanged
// ... enforce
```

**Default mode = `'read'`** so any caller that doesn't opt in stays safe. Only the three write call-sites get updated to pass `mode: 'write'`.

## Warning channel (new)

Today `ValidationResult` is error-only. 0.6.7 adds a warnings channel.

### Type changes

```typescript
interface ValidationWarning {
  field: string;
  message: string;
  code: 'PRECISION_COARSER_THAN_DECLARED';  // one code today, reusable for future soft-checks
  location: SourceLocation | null;
}

interface ValidationResult {
  valid: boolean;           // false only if errors is non-empty
  errors: ValidationError[];
  warnings: ValidationWarning[];  // NEW тАФ never blocks the write
}
```

### Response surface

When a write produces warnings, they surface in the response envelope:

```json
{
  "ok": true,
  "data": { ... },
  "_meta": {
    "request_id": "...",
    "warnings": [
      {
        "field": "started_at",
        "message": "Value \"2026-04-16\" is day-precision but schema declares store_precision=second",
        "code": "PRECISION_COARSER_THAN_DECLARED"
      }
    ]
  }
}
```

### Ops log

Every warning also emits one structured ops log line (`level: "warn"`), with the same fields plus `doc_id`, `doc_type`, `tool`, `session_id`. Operators see patterns across agents without having to crawl response bodies.

### Reuse

Other soft-validations slot into this channel at zero cost:
- Deprecated fields (`deprecated: "use X instead"`) тАФ future schema evolution
- Length hints (`maxLength`, soft) тАФ future
- Cross-field invariants тАФ future

None in scope for 0.6.7. Infrastructure lands once.

## MCP tool surface changes

### `maad_schema` response

Field descriptors include the three new keys when set:

```json
{
  "name": "started_at",
  "type": "date",
  "format": "YYYY-MM-DDTHH:mm:ss",
  "storePrecision": "second",
  "onCoarser": "warn",
  "displayPrecision": "minute"
}
```

Absent keys are omitted (not `null`). Consumers read `displayPrecision` to decide rendering granularity.

### `maad_validate` response

New optional parameter `includePrecision: boolean` (default false). When true, runs precision audit on historical data and reports violations as informational entries (separate from structural errors). Output structure:

```json
{
  "total": 100,
  "valid": 100,
  "invalid": 0,
  "errors": [],
  "precisionDrift": [   // only present when includePrecision=true
    { "docId": "jrn-2026-001", "field": "timestamp", "declared": "second", "actual": "day" }
  ]
}
```

Never mutates `valid`/`invalid` counts тАФ precision drift is informational, not structural.

### `_meta.warnings[]` on write responses

New field. Empty array omitted. Present on `maad_create`, `maad_update`, `maad_bulk_create`, `maad_bulk_update` when warnings fire.

## Round-trip precision preservation (load-bearing fix)

**Discovered during spec review (jrn-2026-025, agt-codex-cli spot-check).** The validator and schema DSL are useless if MAADB's own readтЖТmodifyтЖТwrite pipeline silently degrades precision. It currently does.

### Root cause

`gray-matter` uses `js-yaml` under the hood. YAML 1.1/1.2 defines `!!timestamp` as a native type, so `2026-04-16T17:20:00Z` in a frontmatter block is parsed into a JS `Date` object, not a string. Downstream code then normalizes Dates to `YYYY-MM-DD` at every touch point:

| Site | Line | Behavior |
|---|---|---|
| `src/writer/serializer.ts` | 66-67 | `value.toISOString().slice(0, 10)` тАФ **write path truncation** |
| `src/extractor/fields.ts` | 49-50 | Same pattern for extraction |
| `src/engine/indexing.ts` | 164-176 | Same pattern for index values |
| `src/engine/reads.ts` | 368-369 | Same pattern for read comparisons |
| `src/engine/writes.ts` | 551-557 | Same pattern for `expectedVersion` comparisons |

**Net effect today:** an agent writes `2026-04-16T17:20:00Z` тЖТ gray-matter parses it as Date тЖТ any update touching the record serializes it back as `2026-04-16`. Precision gone. Silent.

### Fix: keep timestamps as literal strings through the whole pipeline

Configure gray-matter's YAML parser to **not coerce timestamps into Date objects.** The cleanest path is a custom `js-yaml` schema that omits the `timestamp` type, passed to gray-matter via its `engines` option.

**Target: one small module, `src/parser/yaml-engine.ts`:**

```typescript
import yaml from 'js-yaml';

// Standard CORE_SCHEMA minus the !!timestamp type.
// Result: every datetime-looking scalar stays a plain string.
const STRING_PRESERVING_SCHEMA = yaml.CORE_SCHEMA;  // CORE excludes timestamp; JSON_SCHEMA also works

export const yamlEngine = {
  parse: (raw: string) => yaml.load(raw, { schema: STRING_PRESERVING_SCHEMA }),
};
```

Then every `matter(raw)` call uses it:

```typescript
matter(raw, { engines: { yaml: yamlEngine } });
```

Call sites to update (audit via grep for `matter(`):
- `src/scanner.ts:69`
- `src/parser/frontmatter.ts:18`
- `src/schema/loader.ts:44,46`
- `src/registry/loader.ts:47,51`
- `src/git/diff.ts:68`
- `src/git/snapshot.ts:61`
- `src/instance/config.ts` (matter import, find exact line)

Consolidate so every `matter()` call goes through a single `parseMatter(raw)` helper that always passes the string-preserving engine. Prevents regression by new code.

### Surviving `instanceof Date` branches

The existing `instanceof Date` branches in validator, writer, extractor, indexer, and reads stay тАФ but as **defensive backstops**, not primary paths. With the YAML engine change, they should be unreachable in practice; if a Date ever slips in (e.g. a user passes one directly via `maad_create fields: { timestamp: new Date() }`), the serializer converts it to **full ISO millisecond precision**, not truncated:

```typescript
// BEFORE (serializer.ts:66-67)
if (value instanceof Date) {
  return `${key}: ${value.toISOString().slice(0, 10)}`;
}

// AFTER
if (value instanceof Date) {
  return `${key}: ${value.toISOString()}`;  // full ms precision, never truncate
}
```

Apply the same change to the other 4 `.slice(0, 10)` sites. They become consistent: Dates serialize at millisecond precision, strings pass through unchanged.

### Why this belongs in 0.6.7 specifically

Without this fix, the precision contract is a lie. An agent that writes `2026-04-16T17:20:00Z` and another agent that edits any neighboring field would lose the time component through MAADB's own round-trip. The schema-declared `store_precision: second` would fire warnings on MAADB's own rewrites тАФ a self-inflicted false positive.

This is the kind of bug that, once shipped, bakes itself into every dataset. Fix before data exists.

### Acceptance test (add to main test table)

## Edge cases

1. **Update that doesn't touch the date field.** Changed-fields check skips precision validation. Historical `timestamp: "2026-04-15"` stays untouched, no warning. Acceptance test T4.
2. **Update that touches the date field with a still-coarse value.** New value is re-checked against declared precision. Warning fires; write proceeds (if `warn`) or rejects (if `error`).
3. **Bulk write with mixed compliance.** Each record validated independently. `warn` records produce warnings; `error` records produce per-record failures in the bulk response. No whole-batch abort.
4. **Schema migration that tightens precision.** Declaring `store_precision: minute` on a previously-lax field тЖТ historical records stay valid in the index (backward-compat invariant); only future writes enforce. Document this in schema-guide.
5. **Date objects from direct caller input.** If a caller passes `new Date(...)` in `fields`, serializer writes it at full ISO millisecond precision (post-fix). Never truncated. Precision detector treats Date as `millisecond` for validation.
6. **Precision detection on a malformed string.** If string doesn't match any recognized pattern, structural validation (existing regex check) fails first. Precision detector never runs on invalid strings.
7. **YAML timestamp with explicit timezone** (e.g. `2026-04-16T17:20:00+02:00`). After the gray-matter fix, stays as literal string. Precision detection honors the seconds/millis suffix; timezone offset doesn't affect precision classification.

## Non-goals

Explicitly out of scope. Named here so they don't scope-creep the PR.

- **Retrofitting existing coarse records.** Historical data stays as-is. Migration tooling is a separate future concern (goes with 0.9.x schema evolution).
- **Full temporal DSL.** Intervals, recurrence, durations тАФ separate followup if the need emerges.
- **Engine-side rendering.** Engine stays data-only. No formatting, no timezone conversion on read.
- **Consumer lock-in on display.** `display_precision` is a hint, never enforcement. Consumers are free to ignore or override per-user.
- **Number-range / string-length soft-checks.** Infrastructure lands here; actual checks are future followups using the same warning channel.
- **Cross-field temporal invariants** (e.g. `ended_at тЙе started_at`). Future.
- **Per-record override of schema precision.** No escape hatch on individual writes тАФ if the schema says `error`, it errors. Consistency beats per-record leniency.

## Risk / rollout

- **Additive by construction.** No schema that doesn't opt in sees any behavior change. 476 existing tests pass unchanged.
- **Default `on_coarser: warn` means any schema that does opt in gets a gentle signal, not a wall.** Graduate to `error` after cleanup.
- **Warning channel infrastructure is reusable.** Cost amortizes across future soft-validations.
- **Load-bearing invariant is a single gate (`options?.mode !== 'write'`).** One line to audit, one test (T6) that locks it.
- **Estimated effort: 3тАУ3.5 days focused.** Phase 1 (primitives) half a day; Phase 2 (round-trip fix) half to one day тАФ concentrated in the YAML engine swap + touching 5 site fixes; Phases 3+4 (warning channel + DSL/enforcement) are the bulk at ~1.5 days; Phase 5 (docs) half a day.

## Ship gate

This spec MUST ship before brain-app goes live. Rationale:

- brain-app stores per-user session / journal data with timestamps
- Without precision enforcement, LLM agents will default to day-precision and permanently lose moment-of-action information
