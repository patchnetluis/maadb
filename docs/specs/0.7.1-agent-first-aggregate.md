## Problem

A real cross-doctype report — "total hours by customer for March 2026" against casedesk-archive (~15K records across `legacy_case_journal → legacy_case → legacy_account`) required **7 MCP tool calls and 10+ minutes** to execute. The same query should complete in **1 call and ~15ms engine-side**. Full diagnosis and recommendations in `jrn-2026-093`.

Three gaps drove the call-count explosion:

1. **Filter expressiveness (G1).** `maad_query` and `maad_aggregate` filters accept exactly one op per field. Date-range queries (`gte X AND lte Y`) are inexpressible, forcing client-side subtraction hacks.
2. **Ref traversal in `groupBy` (G2).** `maad_aggregate.groupBy` accepts only fields on the grouped doctype. Cross-doctype aggregates like "group journals by the *name* of the account that owns the case" require 3–7 engine calls plus Python-side joins.
3. **Silent response truncation (G3).** `maad_query.limit` accepts arbitrarily large values (observed `limit: 2000` accepted silently). Responses exceeding the MCP client's harness cap (~64KB in Claude Code) are dumped to a temp file with a generic error — the agent sees a partial result and has no contract for the truncation boundary.

A fourth, softer gap surfaced in the same report: the agent did not reach for `maad_aggregate` on first pass, falling through to manual record iteration. Tool listing without trigger-rich descriptions invites this failure mode.

## Scope

**In scope (this release):**

- **R3** — Response safety: `RESPONSE_TOO_LARGE` error with continuation cursor, hard caps on `limit`, server-side projected-size guard.
- **R1** — Multi-hop ref traversal in `maad_aggregate.groupBy`: N-hop chain syntax `"ref_field->ref_field->leaf_field"`.
- **R2** — Range/composite filters on `maad_query` and `maad_aggregate`: `between` shortcut + general array-of-ops form.
- **Agent instruction generators** — `src/maad-md.ts` and `src/claude-md.ts` emit trigger rules for `aggregate` and `join` to close the discovery gap.

**Deferred (future release, not this one):**

- **R4** — Date-part buckets in `groupBy` (`field::year-month`, `::quarter`).
- **R5** — Multiple `groupBy` fields (tuple keys).
- **R6** — Compact response mode (`response_mode: "compact"`).
- **R7** — Filter over ref paths (complement to R1 on the filter side).
- **R8** — `countDistinct` metric.
- **R9** — `having` post-filter on aggregate.

**Parallel track, not version-gated:**

- **R10** — Schema-design skill doc (`_skills/reporting-schema.md` or section in `architect-core.md`). Guidance: on reporting paths, denormalize hot FKs (stable IDs, never labels) onto fact records at ingestion. Ships independently of 0.7.1.

## Non-goals (scope guard, verbatim from jrn-2026-093)

> MAADB is an agent-friendly document store with ergonomic aggregates — not an OLAP engine. If reports regularly drive schema decisions or routinely scan >50K records, the right break is a separate reporting module that consumes MAADB dumps, not bolting analytics features onto MAADB itself.

This language goes verbatim into ROADMAP.md at release and defends against future pressure to add:

- Time-window materialized views
- User-defined computed fields
- Multi-dimensional pivots beyond a single `groupBy` column
- Query-plan optimizers or custom index types

Each of those is a reporting-engine feature. MAADB's mandate stops at "cross-doctype aggregate over indexed FKs in one call."

## R3 — Response Safety

### Problem

`maad_query.limit` accepts arbitrarily large values. When the serialized response exceeds the MCP client's tool-output cap, Claude Code writes the result to a temp file and returns a generic error. The agent sees an incomplete read with no contract for "how much is there." Silent truncation of this shape is the worst class of bug — subsequent reasoning is based on data the agent believes it has but doesn't.

### Phase split (scope adjustment 2026-04-22)

R3 lands in two phases:

- **Phase A — 0.7.1 (this release):** hard caps on `limit` + projected-size guard + `_meta.limit_clamped`. Addresses the core safety concern — no silent truncation, size cap actionable. **Shipped.**
- **Phase B — 0.7.2 (follow-up):** cursor continuation on `maad_query` and `maad_aggregate`, `CURSOR_INVALID` error, backend SQL pagination. Ergonomic addition on top of Phase A's safety floor. Tracked in the brain as a separate followup. Not blocking for 0.7.1's shipping criteria.

The design below specifies the full R3 surface; Phase A items are implemented, Phase B items are deferred verbatim.

### Design

**1. Hard caps at accept time.**

| Tool | Field | Cap | Behavior on over-request |
|---|---|---|---|
| `maad_query` | `limit` | 500 | clamp, set `_meta.limit_clamped` |
| `maad_aggregate` | `limit` | 2000 groups | clamp, set `_meta.limit_clamped` |

Clamping is forgiving — requesting 10000 yields 500 rows with a `_meta` note, not an error. Consistent with how `changes_since` already treats its `limit` parameter (clamp at 1000).

**2. Server-side projected-size guard.**

After the result array is built but before JSON serialization returns to the client, measure `JSON.stringify(result).length` (in bytes). If it exceeds the configurable cap (default **64 KB**, env `MAAD_RESPONSE_MAX_BYTES`), return:

```json
{
  "ok": false,
  "errors": [{
    "code": "RESPONSE_TOO_LARGE",
    "message": "Projected response (94KB) exceeds cap (64KB)",
    "hint": "Narrow field projection, add filters, or paginate with cursor",
    "projectedBytes": 94208,
    "capBytes": 65536
  }]
}
```

**3. Cursor continuation.**

Both `maad_query` and `maad_aggregate` gain an optional `cursor` parameter and return `nextCursor` when results are incomplete. Cursor shape: opaque base64url, reuses the `changes_since` pattern:

- **Query:** cursor encodes `(sortField, sortValue, docId)` — stable tuple, monotonic pagination.
- **Aggregate:** cursor encodes `(groupKey)` in the result's native sort order — typically alphabetical on the group field.

Callers resume by passing `cursor: <last-nextCursor>`. Empty cursor = start from beginning. This makes pagination safe under concurrent writes — new records appearing mid-pagination simply extend the tail, never disrupt the cursor contract.

**4. `_meta.limit_clamped` shape.**

```json
{
  "data": [...],
  "_meta": {
    "limit_clamped": { "requested": 10000, "applied": 500 }
  }
}
```

Omitted when not clamped. Agents that ignore `_meta` see zero behavioral change beyond the smaller result set.

### Error taxonomy — R3

| Code | When |
|---|---|
| `RESPONSE_TOO_LARGE` | projected serialized response exceeds `MAAD_RESPONSE_MAX_BYTES` |
| `CURSOR_INVALID` | cursor failed base64url decode or tuple shape mismatch |
| `CURSOR_STALE` | cursor references a state the index no longer supports (e.g. schema change between pages) — caller should restart from empty cursor |

### Test matrix — R3

- **Hard cap clamp** — request `limit: 10000` on query and aggregate, verify response has ≤ cap rows + `_meta.limit_clamped`.
- **Projected-size reject** — stage a response projected > 64KB, verify `RESPONSE_TOO_LARGE` with correct `projectedBytes` / `capBytes`.
- **Cursor round-trip** — paginate through a 1500-row corpus in 3 pages of 500, verify no duplicates and no skipped records.
- **Cursor under concurrent insert** — paginate while new records are inserted mid-stream; verify stable pagination (new records appear on page 4, never disrupt pages 1–3).
- **Invalid cursor** — malformed base64url and wrong-shape cursors both return `CURSOR_INVALID`.

## R1 — Multi-Hop Ref Traversal in `groupBy`

### Problem

`maad_aggregate.groupBy` accepts a field name on the grouped doctype. Cross-doctype aggregates (group journals by the name of an account two hops away) require client-side join logic.

### Design

**Grammar.** `groupBy` accepts either a single field name (existing behavior) or a hop chain:

```
groupBy: "legacy_case_id->legacy_account_id->name"
```

Syntax: `<ref-field>->(<ref-field>->)*<leaf-field>`. Arbitrary depth N, no arbitrary upper bound (indexed FK lookups are cheap; at casedesk scale 30 groups × 2 hops = 60 O(1) lookups, microseconds).

**Resolution semantics.**

1. Parse `groupBy` by `->`. Validate each segment against the schema chain at load time (`SCHEMA_REF_CHAIN_INVALID` on failure).
2. For each record in the aggregation corpus:
   a. Read the first hop's ref field → target docId.
   b. Follow the ref through the target's schema to the next hop's ref field.
   c. Repeat until the leaf segment, which is a non-ref field (string/enum/number).
   d. The leaf value is the group key.
3. Records where any hop resolves to `null` or a missing target are binned under a synthetic `__unresolved__` group key. This is visible to the caller so data quality issues surface rather than silently vanish.

**Why N-hop from the start, not one-hop minimum.**

The per-hop loop is the same code whether N=1, N=2, or N=5. No additional complexity. Tests cover 1-hop, 2-hop, and the `__unresolved__` path explicitly; deeper hops are a scale concern (unlikely in practice but not disallowed).

**Combined with R10 (denorm hot FKs).** Agents following R10's schema guidance store `legacy_account_id` directly on `legacy_case_journal` at ingestion. Then:

```
groupBy: "legacy_account_id->name"
```

is one-hop, zero join cost at query time, and works whether the schema authors adopted R10 or not.

### Error taxonomy — R1

| Code | When |
|---|---|
| `SCHEMA_REF_CHAIN_INVALID` | `groupBy` chain references a field that doesn't exist or isn't a ref (at load time) |
| `REF_CHAIN_BROKEN` | at query time, a ref resolves to a missing target (logged once per aggregate call, record binned under `__unresolved__`) |

Note: `REF_CHAIN_BROKEN` is a soft signal, not a failure — the query completes and returns results, with the unresolved records visible as a group.

### Test matrix — R1

- **One-hop happy path** — `groupBy: "legacy_case_id->status"` over a 3-doctype corpus, verify group counts match direct SQL join.
- **Two-hop happy path** — the canonical jrn-2026-093 report: journals → cases → accounts, grouped by account name.
- **Chain validation at load** — malformed chain syntax, non-ref in middle segment, ref to non-existent target, all fail at schema load with `SCHEMA_REF_CHAIN_INVALID`.
- **Broken ref at runtime** — record with `legacy_case_id` pointing to a deleted case, verify binned under `__unresolved__` without killing the query.
- **Deep chain (3-hop)** — verify no special-case behavior at N=3; same semantics as 1 and 2.
- **Combined with R10 pattern** — denorm'd `legacy_account_id` directly on journal, one-hop resolution, verify zero cross-doctype SQL in the query plan.

## R2 — Range / Composite Filters

### Problem

Filter spec accepts `{ field: value }` (implicit eq) or `{ field: { op, value } }` (single op). Expressing a date range requires two separate queries and client-side subtraction.

### Design

**Two forms, both supported.** The common case gets a shortcut; the general case gets array-of-ops.

**Form 1 — `between` shortcut (common case):**

```json
{ "created_at": { "op": "between", "value": ["2026-03-01", "2026-03-31"] } }
```

- `value` is a 2-tuple `[lo, hi]`.
- Inclusive on both ends.
- Type-agnostic — works for dates, numbers, strings lexically.

**Form 2 — array-of-ops (general case):**

```json
{ "created_at": [
    { "op": "gte", "value": "2026-03-01" },
    { "op": "lte", "value": "2026-03-31" }
  ]
}
```

- All ops in the array combine with AND semantics.
- Any valid filter op works inside the array (eq, neq, gt, gte, lt, lte, in, contains).
- Empty array is rejected (`FILTER_EMPTY_ARRAY`).
- Mixing ops is allowed (`[{op: "gte"}, {op: "neq"}]`).

**`between` desugars to array-of-ops internally** so the evaluator has one code path. `{field: {op: "between", value: [a, b]}}` becomes `{field: [{op: "gte", value: a}, {op: "lte", value: b}]}` at normalize time.

**Applies identically to `maad_query` and `maad_aggregate` filter params.**

### Error taxonomy — R2

| Code | When |
|---|---|
| `FILTER_BETWEEN_INVALID` | `between` value is not a 2-tuple, or `lo > hi` |
| `FILTER_EMPTY_ARRAY` | array-of-ops filter is empty |
| `FILTER_OP_INVALID` | op inside array or single form is not in the known set |

### Test matrix — R2

- **Between inclusive bounds** — date range hits records exactly at both endpoints.
- **Between with `lo > hi`** — rejected at parse time.
- **Array-of-ops AND semantics** — `gte + lte` matches only records in the intersection; `gte + neq` excludes the specific value.
- **Mixed with other fields** — multi-field query where one field uses `between` and another uses a scalar filter.
- **Empty array** — rejected.
- **Aggregate with range filter + R1 groupBy** — the canonical jrn-2026-093 report shape.

## Agent-instruction Generator Updates

### Problem

`maad_aggregate` and `maad_join` are listed in MAAD.md's Read command table with one-line definitions but no trigger rules. Compare to `search` (Rule 5) and `related` (Rule 6), which have explicit "when tempted to do X manually, use Y instead" rules. Agents miss aggregate/join on first pass because nothing pushes them toward it.

### Design — `src/maad-md.ts`

Add to the Rules section (currently rules 1–9, already includes `_meta.warnings[]` inspection):

```
10. **Use `aggregate` for group-by totals** — don't iterate records to compute counts, sums, averages, mins, maxes. `maad_aggregate` with the metric and `groupBy` returns the answer in one call.
11. **Use `join` to follow refs in one call** — don't chain `query` → `get` → `get` to collect related fields. `maad_join` projects fields from both sides of a ref in one pass.
```

Sharpen the Read command table descriptions (exact text change):

- `aggregate` — from *"Group by a field with optional metric (count/sum/avg/min/max)"* to *"Group by a field with count/sum/avg/min/max. **Use instead of iterating records.** Supports multi-hop ref chains (`a->b->c`) for cross-doctype aggregates."*
- `join` — from *"Query + follow refs + project fields from both sides in one call"* to *"Query + follow refs + project fields from both sides in one call. **Use instead of query → get → get chains.**"*

### Design — `src/claude-md.ts`

Add to the Rules section (currently rules 1–6):

```
7. Use `maad_aggregate` for group-by totals — don't page through records to compute counts, sums, or averages.
8. Use `maad_join` to follow refs and collect fields in one call — don't chain query/get loops.
```

Sharpen the Reading data list:

- `maad_aggregate` — from *"count/sum/avg/min/max grouped by a field"* to *"count/sum/avg/min/max grouped by a field, including multi-hop ref chains. Use instead of iterating records for totals."*
- `maad_join` — from *"query + follow refs + project fields from both sides in one call"* to *"query + follow refs + project fields from both sides in one call. Use instead of query→get→get chains."*

### Propagation

- **MAAD.md** — regenerates on every `reindex` in every project. Propagates automatically post-release.
- **CLAUDE.md** — init-only; existing projects keep their pre-0.7.1 CLAUDE.md. Documented limitation; users can regenerate manually if they want the update. Not auto-overwriting is deliberate — prevents clobbering hand-edits.

## Backward compatibility

All four changes are additive:

- **R3** — existing clients that pass `limit ≤ 500` see no change. Agents that ignore `_meta` keep working; they just see clamped results instead of truncation. `nextCursor` is optional — absent on complete responses.
- **R1** — existing single-field `groupBy` keeps working unchanged. The `->` syntax is only recognized when present.
- **R2** — existing scalar filters and `{op, value}` single-op filters keep working. `between` and array forms are new recognized shapes.
- **Generator rules** — MAAD.md rebuild is idempotent; CLAUDE.md isn't regenerated in existing projects at all.

No breaking changes. No migration required. Full 0.7.0 test suite must remain green.

## Canonical single-call report (golden test)

The jrn-2026-093 report becomes one call:

```json
{
  "tool": "maad_aggregate",
  "args": {
    "docType": "legacy_case_journal",
    "filters": {
      "created_at": { "op": "between", "value": ["2026-03-01", "2026-03-31"] }
    },
    "groupBy": "legacy_case_id->legacy_account_id->name",
    "metric": { "field": "time_hours", "op": "sum" }
  }
}
```

Returns `[{ key: "Acme Corp", value: 47.5, count: 12 }, ...]`. One call. ~15ms against casedesk-archive's 15K records. End of report.

## Out of scope (deferred, explicit)

- **R4 date-part buckets** — useful for time-series pivots but not blocking the canonical report.
- **R5 multi-field `groupBy`** — pivots (customer × month) are a follow-on feature.
- **R6 compact response mode** — nice-to-have size reduction; deferred until R3's projected-size guard shows real pressure.
- **R7 filter over ref paths** — complement to R1, only needed if "find all journals belonging to account X" becomes a hot path.
- **R8 `countDistinct`** — common but not blocking.
- **R9 `having` post-filter** — deferred until multi-metric use cases surface.

All of the above can be revisited post-0.7.1 based on demand signal.
