# 0.4.1 — Per-Request Timeouts + Graceful Shutdown

Spec for bounded request execution and a clean SIGTERM path. Ensures container restarts never leave stale git locks, in-flight writes either complete or reject deterministically, and the engine exits in a known-good state.

## Problem

Today:

1. **No request timeout.** A tool call that hangs (slow filesystem, stuck git subprocess, pathological extraction) holds its slot indefinitely. Under rate-limit concurrent caps, a stuck request starves other callers.
2. **Abrupt shutdown.** `registerShutdownHooks` in `src/mcp/lifecycle.ts` calls `engine.close()` and `process.exit(0)` on SIGTERM/SIGINT. In-flight writes can exit mid-operation, leaving:
   - File on disk written but not indexed (journal catches on next boot).
   - File indexed but not git-committed (`.git/index.lock` potentially left over).
   - Idempotency / rate-limit state dropped with no warning to clients.
3. **No way to reject new work during shutdown.** New tool calls during the shutdown window either race with the exit or succeed and then get dropped.

Container orchestrators (Docker, Azure Container Apps, Kubernetes) send SIGTERM and expect a graceful window (default 10s on Docker, 30s on K8s) before SIGKILL. MAADB must use that window deliberately.

## Goals

- Every MCP tool call has a bounded maximum duration. Exceeding it returns a structured error.
- SIGTERM triggers a drain: new tool calls are rejected, in-flight calls are given a bounded window to complete, then the process exits.
- Shutdown completes in-flight writes cleanly (mutex drains, journal settles, git commits land, SQLite closes, audit log flushes).
- No `.git/index.lock` ever left from a clean shutdown. (Dirty shutdowns — SIGKILL, crashes — are the H3 recovery story.)
- Tunable shutdown grace window for operators with non-default orchestrator timeouts.

## Non-goals

- Active cancellation of in-flight writes. If a write is already running, the drain waits for it; it does not interrupt. Cancellation mid-write risks partial state worse than the overrun.
- Work queue persistence across restarts. In-flight requests that haven't started executing are rejected with `SHUTTING_DOWN`; the client is responsible for retry post-restart.
- Kubernetes-style preStop hooks in-engine. Operators wire that up externally.

## Design

### Per-request timeout

Every tool call runs under a timeout ceiling, enforced in `withSession` / `withEngine`:

```ts
const TIMEOUT_MS = Number(process.env.MAAD_REQUEST_TIMEOUT_MS ?? 30_000);

const response = await Promise.race([
  toolHandler(),
  new Promise<ToolResponse>((_, reject) => setTimeout(
    () => reject(new RequestTimeoutError(toolName, TIMEOUT_MS)),
    TIMEOUT_MS,
  )),
]);
```

On timeout:
- Response: `{ ok: false, errors: [{ code: 'REQUEST_TIMEOUT', message: ..., details: { tool, limitMs } }] }`
- Behavior depends on *where* the request was when the timeout fired — see Cooperative Cancellation below.
- Concurrent-slot release (from rate-limit H5) fires when the tool handler actually completes, not on the timeout — otherwise a timed-out but still-running write could race against a newly admitted one.

Default 30s. Tunable via `MAAD_REQUEST_TIMEOUT_MS`. Bulk operations may need a higher limit — operators set it explicitly.

### Cooperative cancellation

Cancellation is *cooperative at stage boundaries*, not preemptive. The handler is a chain of async stages (mutex wait → file write → index → git commit → response). An `AbortSignal` is threaded from the timeout into the handler; each stage boundary checks it before starting the next step. Once a stage is in motion, it completes.

Three bands of behavior:

| Request state at timeout | Action | Rationale |
|---|---|---|
| Queued in mutex (not yet holding lock) | Remove from queue, return `REQUEST_TIMEOUT`, never run engine method | Safe: no side effects to unwind. Drop cleanly. |
| Holding mutex, between stages | Let current stage complete, then check signal, then return `REQUEST_TIMEOUT` to caller. Engine state is consistent at that stage boundary. | Partial work is not rolled back, but every stage boundary is a valid stopping point — journal, file on disk, backend row — and the startup-recovery logic already knows how to finish partially-committed writes on the next boot. |
| Inside a single stage (e.g. mid git commit) | Stage runs to completion. Response returns `REQUEST_TIMEOUT`. Completed work stands. Delayed `tool_call_overrun` log line on stage completion. | Git commits, SQLite transactions, and `fsync` are not safely interruptible at the OS level. Cancelling mid-operation risks partial state worse than the overrun. Cooperative checkpoints at stage boundaries are the right granularity. |

**What cooperative cancellation does NOT do:**

- It does not cancel an in-flight git commit.
- It does not roll back a file that has been written to disk.
- It does not remove a backend index row that has been inserted.

The client receives `REQUEST_TIMEOUT` when the timeout fires, but the server may still finish the write. This is the trade-off of "safe cancellation" vs "fast response." For write-heavy clients, the durable side-effect has already happened by the time the timeout fires; the subsequent retry (with an idempotency key) replays the cached response rather than creating a duplicate. This is precisely why H4 exists.

**Implementation seam for AbortSignal:** add an optional `signal?: AbortSignal` parameter to mutating engine methods. At each stage boundary (after mutex acquire but before file write, after file write but before index, after index but before git), check `signal.aborted`. If aborted and no irreversible work has started in this stage, return `REQUEST_CANCELLED` internally (promoted to `REQUEST_TIMEOUT` at the wrapper). Existing callers that pass no signal see no change.

### Shutdown state machine

```
RUNNING  --SIGTERM-->  DRAINING  --drain complete or timeout-->  EXITING  -->  exit(0 or 1)
```

Three states; tracked by a single module-level variable in `src/mcp/lifecycle.ts`:

```ts
let state: 'running' | 'draining' | 'exiting' = 'running';
```

#### RUNNING

Normal operation. All checks pass through as designed.

#### DRAINING

Triggered by first SIGTERM or SIGINT. Transitions:

1. Flip `state` to `'draining'`.
2. Emit ops log line `event: 'shutdown_start'`.
3. Reject new tool calls. `withSession` checks `state` at entry — if `draining` or `exiting`, return `{ ok: false, errors: [{ code: 'SHUTTING_DOWN' }] }` immediately, before any other check.
4. Wait for in-flight writes to finish. Specifically, wait for:
   - The engine write mutex to become idle (`writeLock.depth() === 0`).
   - All in-flight concurrent-slot holders to release (rate limiter's `concurrent` map empty across all sessions).
5. Apply the grace window — if the drain doesn't complete within `MAAD_SHUTDOWN_TIMEOUT_MS` (default 10s), force-exit into EXITING and emit a `shutdown_timeout` warning log line.

Step 4's wait is a polled loop with 50ms backoff. Cheap; not time-critical.

#### EXITING

Final cleanup, always runs whether drain succeeded or timed out:

1. Close the engine (`engine.close()` — closes SQLite backend cleanly via existing logic).
2. Flush the audit log (`auditLog.flush()` if pino; relevant for async-mode loggers).
3. Dispose per-session rate-limit state.
4. Emit ops log `event: 'shutdown_complete'` with `elapsed_ms` since shutdown_start and a boolean `drain_timed_out`.
5. `process.exit(drainTimedOut ? 1 : 0)`.

Exit code 1 on timed-out drain surfaces the issue to operators monitoring container exit codes. Clean drain is 0.

### Second SIGTERM / SIGINT

If another signal arrives while `state === 'draining'`: accelerate. Skip the rest of the drain wait, jump directly to EXITING with a `shutdown_forced` log line. Operators who mean it get to override.

A third signal during EXITING: ignore; we're already exiting.

### Mutex interaction

The drain explicitly waits for `writeLock.depth() === 0`. Since `SHUTTING_DOWN` rejects new calls at the wrapper level, nothing new can enter the mutex during drain. So the wait terminates as soon as all currently-queued writes complete.

Edge case: a write currently blocked on the mutex (queued behind another) when SIGTERM arrives. Two options:

- **Option A: Let queued writes execute.** They're already accepted; rejecting now loses the client's request. The drain waits for all of them.
- **Option B: Reject queued writes.** Fail fast, client retries after restart.

Choice: **Option A**. Rationale: the client has the response in flight and expects completion; bailing out creates a "is my write saved?" ambiguity that defeats the point of idempotency keys. If operators need faster drain, they set a tighter `MAAD_SHUTDOWN_TIMEOUT_MS`.

### Idempotency and rate-limit state

Both are in-memory and discarded at exit. Documented behavior:

- Post-restart idempotency-key retries run as fresh writes (no cache match).
- Post-restart rate-limit buckets are refreshed (all clients get a full burst quota).

For H4 clients who need cross-restart idempotency, that's a larger storage question deferred to a future version.

### Error codes

```ts
| 'REQUEST_TIMEOUT'    // per-request timeout exceeded
| 'SHUTTING_DOWN'      // server is draining or exiting; retry elsewhere
```

`SHUTTING_DOWN` is already reserved in the H1 (concurrency) spec's error code section; this spec is the module that emits it.

### Env knobs

| Var | Default | Meaning |
|---|---|---|
| `MAAD_REQUEST_TIMEOUT_MS` | 30000 | per-request timeout ceiling |
| `MAAD_SHUTDOWN_TIMEOUT_MS` | 10000 | drain grace window before forced exit |

## Test plan

`test/mcp/lifecycle.test.ts`:

### S1 — Request timeout fires (mid-stage)
Register a deliberately slow tool handler (35s). Call it with `MAAD_REQUEST_TIMEOUT_MS=1000`. Response returns within ~1.1s with `REQUEST_TIMEOUT`. Ops log contains a `tool_call_overrun` line when the slow handler eventually completes. Handler's side effects persist (this verifies the "completed work stands" contract).

### S1b — Request timeout fires while queued in mutex
Hold the mutex with a slow write. Enqueue a second write behind it. Fire timeout on the queued write before the first completes. Assert: queued write returns `REQUEST_TIMEOUT` *without* the engine method ever running. Backend has only the first doc.

### S2 — Graceful drain completes
Start an engine. Issue a slow `createDocument` (stubbed to take 2s). While it runs, send `SIGTERM`. Assert:
- State transitions: `running` → `draining` → `exiting`.
- The `createDocument` completes successfully.
- Process exits with code 0.
- Ops log contains `shutdown_start`, `shutdown_complete` with `drain_timed_out: false`.

### S3 — Drain timeout forces exit
Slow handler (30s). Set `MAAD_SHUTDOWN_TIMEOUT_MS=500`. SIGTERM. Assert:
- Process exits with code 1 within ~600ms.
- Ops log contains `shutdown_timeout` warning.

### S4 — New requests during drain are rejected
SIGTERM while a slow write is in-flight. Attempt another `maad_create` 100ms later. Response: `{ ok: false, errors: [{ code: 'SHUTTING_DOWN' }] }`.

### S5 — Second SIGTERM accelerates
SIGTERM, then 100ms later another SIGTERM. Process exits immediately without waiting for drain completion. Log contains `shutdown_forced`.

### S6 — Queued writes execute on drain (Option A verification)
Set `writes/sec=1`. Send 3 writes rapidly; two queue in the mutex. SIGTERM after the first starts. All three complete successfully before exit.

### S7 — Clean git state on exit
Issue a write, SIGTERM immediately after it starts. After drain completes, assert `.git/index.lock` does not exist under the project directory.

### S8 — Engine close is idempotent
Explicitly call `engine.close()`, then trigger shutdown. No errors; second close is a no-op.

## Interactions

- **Mutex (H2):** drain loop waits on `writeLock.depth() === 0`. Queued writes complete before exit.
- **Stale git lock (H3):** clean shutdown path never leaves a lock. Dirty shutdowns (SIGKILL, crash) are H3's responsibility on next boot.
- **Idempotency (H4):** cache discarded at exit; documented.
- **Rate limit (H5):** drain waits on `totalInFlight() === 0`; per-session state discarded at exit.
- **Logging (H6):** audit log flushed before exit; ops log emits shutdown lifecycle events; timed-out tool calls emit a delayed `tool_call_overrun` line on completion.
