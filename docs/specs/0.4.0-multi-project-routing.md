# MAADB v0.4.0 — Multi-Project Routing

**Status:** spec
**Target release:** 0.4.0
**Depends on:** 0.2.13 (empty-project boot)
**Enables:** 0.4.5 (deploy workflow), 0.9.0 (remote MCP over HTTP/SSE)

## Goal

One MCP server process serves many MAAD projects. Each MCP client connection is a session; sessions bind to a project (single mode) or a project whitelist (multi mode) and cannot escape that binding mid-life. Foundation for HTTP/SSE in 0.9.0.

## Non-goals

- Mid-session mode switching (end and reconnect)
- Pool eviction policy (seam only; policy deferred to 0.9.0)
- Cross-project queries or joins (each call hits exactly one project)
- Web UI, remote transport, auth tokens (all 0.9.0+)

---

## Architecture

```
┌─ MCP Server process ─────────────────────────────┐
│  CLI args → Instance (loaded from instance.yaml)│
│                                                  │
│  Instance                                        │
│    ├─ EnginePool       Map<projectName, Engine>  │
│    └─ SessionRegistry  Map<sessionId, State>     │
│                                                  │
│  Tools registered once; dynamic visibility +     │
│  per-call routing via withSession(ctx, req, fn)  │
└──────────────────────────────────────────────────┘
```

## `instance.yaml`

```yaml
name: patchnet-brain-instance
projects:
  - name: brain
    path: C:/Dev/maadb-patchnet-ai-brain
    role: admin
    description: Patchnet-internal knowledge base
  - name: maadb
    path: C:/Dev/maadb
    role: reader
```

**Fields:**
- `name` (required, string) — instance label (used in logs/diagnostics)
- `projects` (required, array, min 1)
  - `name` (required, slug) — used in `maad_use_project <name>`
  - `path` (required, absolute path) — project root
  - `role` (optional, `reader | writer | admin`, default `reader`) — max role granted to sessions binding this project
  - `description` (optional, string) — shown in `maad_projects` list

**Validation errors** surface at startup, before any MCP connection accepts. Invalid file = fail fast.

### Legacy `--project` path

`--project <path>` (or `MAAD_PROJECT` env) synthesizes a one-project instance internally:

```yaml
name: <basename(path)>-legacy
projects:
  - name: default
    path: <path>
    role: <from --role or default reader>
```

Sessions auto-bind single mode to `default` on connect. Behaviorally identical to today: client sees only tools for that role, no `maad_use_project` required.

## SessionState

```ts
type SessionMode = 'single' | 'multi';
type Role = 'reader' | 'writer' | 'admin';

interface SessionState {
  sessionId: string;
  mode: SessionMode | null;      // null = unbound
  activeProject?: string;        // single mode only
  whitelist?: string[];          // multi mode only
  effectiveRoles: Map<string, Role>;  // per-project final role after downgrade
  createdAt: Date;
  lastActivityAt: Date;
}
```

### Session ID resolution

- **stdio (today)**: one client per process. Session ID derived from MCP SDK's request context (`request._meta` or equivalent). If SDK does not supply one, generate a process-lifetime UUID and reuse it.
- **HTTP/SSE (0.9.0)**: session ID from transport layer (`StreamableHTTPServerTransport` header / query).

Design rule: tools never hardcode "there is one session" — always pull from the registry keyed by request context. Enables drop-in HTTP support in 0.9.0.

### Lifecycle

1. **Connect** — MCP `initialize` request → `SessionRegistry.create(sessionId)` → mode `null`. Legacy single-project mode auto-binds at this step.
2. **Bind** — client calls `maad_use_project <name>` or `maad_use_projects [names...]`. Validates names against instance + role against `as` param. Sets mode.
3. **Tool call** — `withSession` resolves engine, checks role, invokes handler.
4. **Disconnect** — `SessionRegistry.destroy(sessionId)`. Does not evict pool entries.

## Routing rules (`withSession`)

Pseudocode:

```ts
async function withSession<T>(
  ctx: InstanceCtx,
  req: McpRequest,
  toolName: string,
  minRole: Role,
  handler: (engine: MaadEngine, projectName: string) => Promise<T>
): Promise<T> {
  const state = ctx.sessions.get(resolveSessionId(req));
  if (!state.mode) throw mcpError('SESSION_UNBOUND', 'call maad_use_project(s) first');

  let projectName: string;
  if (state.mode === 'single') {
    projectName = state.activeProject!;
  } else {
    projectName = req.params.project;
    if (!projectName) throw mcpError('PROJECT_REQUIRED', 'pass project= in multi mode');
    if (!state.whitelist!.includes(projectName)) throw mcpError('PROJECT_NOT_WHITELISTED');
  }

  const role = state.effectiveRoles.get(projectName)!;
  if (!roleSatisfies(role, minRole)) throw mcpError('INSUFFICIENT_ROLE');

  const engine = await ctx.pool.get(projectName);  // lazy init + ensureProjectSkills
  return handler(engine, projectName);
}
```

## Tool registration refactor

### Before (today)

```ts
export function register(server: McpServer, engine: MaadEngine) {
  server.tool('maad_get', ..., async (args) => {
    return engine.getDocumentFull(args.docId);
  });
}
```

Engine is baked into closure at startup.

### After

```ts
export function register(server: McpServer, ctx: InstanceCtx) {
  server.tool('maad_get', ..., async (args, extra) => {
    return withSession(ctx, extra.request, 'maad_get', 'reader', async (engine) => {
      return engine.getDocumentFull(args.docId);
    });
  });
}
```

Every tool resolves its engine per-call from the session.

### Dynamic tool visibility

MCP SDK supports server-initiated tool list notifications (`notifications/tools/list_changed`). On session bind, server re-emits the tool list filtered by the session's max effective role:

- Unbound session (in instance mode): sees only the 4 instance-level tools + `maad_projects`.
- Bound single-mode: sees tools allowed by `effectiveRoles[activeProject]`.
- Bound multi-mode: sees union of tools allowed by any whitelisted project's effective role.

Filtering avoids the "listed but always denied" UX issue.

## The 4 instance-level tools

| Tool | Role | Purpose |
|------|------|---------|
| `maad_projects` | unbound-ok | Lists projects in instance (name, description, whether session has access). |
| `maad_use_project <name> [as=<role>]` | unbound-ok | Binds session to single mode. `as` optionally downgrades role; must not exceed project's configured role. |
| `maad_use_projects [names...] [as=<role>]` | unbound-ok | Binds session to multi mode with a whitelist. Same `as` rule applies to all listed projects. |
| `maad_current_session` | any-bound | Returns session state (mode, projects, effective roles) for debugging. |

`maad_use_project(s)` is one-shot — calling it when already bound returns an error telling the client to disconnect and reconnect.

---

## Role downgrade (`as` param)

- `maad_use_project brain as=reader` when `instance.yaml` has `role: admin` → session bound with effective role `reader` for `brain`.
- `maad_use_project brain as=admin` when `instance.yaml` has `role: reader` → error `ROLE_UPGRADE_DENIED`.
- `as` omitted → effective role = configured role.

Stored in `SessionState.effectiveRoles`. Tool filtering + `withSession` role check both key off this map, so downgrade is enforced uniformly.

## Empty-project boot in the pool

`EnginePool.get(name)`:

```ts
async get(name: string): Promise<MaadEngine> {
  if (this.engines.has(name)) return this.engines.get(name)!;
  const project = this.instance.getProject(name);
  const engine = new MaadEngine();
  await engine.init(project.path);          // self-heals registry/schema/backend
  ensureProjectSkills(project.path);         // scaffolds _skills/ if missing
  this.engines.set(name, engine);
  return engine;
}
```

First call to any tool against a new project triggers init + skills scaffold. Same guarantees as 0.2.13 single-project boot; no pre-flight step needed for `maad_use_project` to succeed against empty dirs.

## Pool eviction seam

```ts
class EnginePool {
  async evict(name: string): Promise<void> {
    const engine = this.engines.get(name);
    if (!engine) return;
    await engine.close();   // close SQLite, flush journal
    this.engines.delete(name);
  }
}
```

**No policy in v1.** `evict` is public but unused by the pool itself. 0.9.0 layers LRU or TTL on top by calling `evict` from a scheduler.

## Backward compatibility

| Scenario | Behavior |
|----------|----------|
| `maad serve --project X --role reader` (today) | Synthetic instance, session auto-binds single mode, reader tools visible. Byte-for-byte compatible with 0.2.x client behavior. |
| `MAAD_PROJECT=X maad serve` | Same as above, default role reader. |
| `maad serve` with no flags | Error: "provide --project or --instance (or set env var)". |
| Both `--project` and `--instance` | Error: mutually exclusive. |

No existing Claude Code / Claude Desktop MCP config needs to change. Upgrading to 0.4.0 is a drop-in; multi-project is opt-in by switching to `--instance`.

## Testing plan

- **Unit**
  - `instance/config.ts` — valid yaml, missing fields, role validation, duplicate project names
  - `instance/pool.ts` — lazy init, second `get` returns cached engine, `evict` closes SQLite, `ensureProjectSkills` runs once
  - `instance/session.ts` — create/destroy, mode transitions, whitelist enforcement, role downgrade math

- **Integration**
  - Two-project instance: bind A single mode, verify B is invisible and unreachable
  - Multi mode: whitelist [A,B], calls without `project=` fail, calls with `project=C` fail
  - Role downgrade: project configured admin, session bound as reader, writer tool denied
  - Empty project: `maad_use_project` against empty dir triggers pool init + skills scaffold
  - Legacy `--project` path: full 0.2.x behavior regression pass

- **Tool filtering**
  - Unbound session sees only instance tools
  - Reader-effective session does not list `maad_create`
  - After `maad_use_project`, `notifications/tools/list_changed` fires once with filtered set

## Error taxonomy (new MCP error codes)

| Code | When |
|------|------|
| `SESSION_UNBOUND` | tool call before `maad_use_project(s)` |
| `SESSION_ALREADY_BOUND` | second `maad_use_project(s)` in same session |
| `PROJECT_REQUIRED` | multi mode call without `project=` arg |
| `PROJECT_NOT_WHITELISTED` | multi mode `project=` outside whitelist |
| `PROJECT_UNKNOWN` | name not in `instance.yaml` |
| `INSUFFICIENT_ROLE` | tool requires role above session's effective role |
| `ROLE_UPGRADE_DENIED` | `as=` asks for role above project's configured role |
| `INSTANCE_CONFIG_INVALID` | startup-only; server refuses to start |
