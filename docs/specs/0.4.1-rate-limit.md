# 0.4.1 — Per-Session Rate Limiting

Spec for a per-session token bucket that caps incoming request rate and payload size. Protects the engine from runaway agents and flood scenarios.

## Problem

An agent in a retry loop or a buggy automation can hammer the MCP server unbounded. Today there is no ceiling — the only back-pressure is whatever the transport imposes (none, for stdio). With HTTP/SSE (0.5.0) arriving and container-hosted deployments on the horizon, one bad client must not be able to saturate an engine's write mutex or exhaust process memory by sending huge payloads.

## Goals

- Bound request rate per MCP session, independent of transport.
- Bound single-request payload size to prevent memory exhaustion from oversize tool args.
- Give callers a clean, machine-readable rejection when they exceed limits — not a crash, not a hang.
- Keep the limits tunable via env so operators can adjust without code changes.
- Zero cost for well-behaved clients — no latency from the limiter on the happy path.

## Non-goals (deferred to 0.8.5)

- Per-token or per-tier rate limits (different limits for different tokens). 0.4.1 applies the same limits to every session.
- Global engine-wide rate limit across all sessions.
- Queue-depth / backpressure signals distinct from rate-limit rejections.
- Burst credit accumulation across idle periods longer than 1 minute (the bucket is simple).

## Design

### Token bucket, per session

Each session gets three independent token buckets:

| Bucket | Capacity (burst) | Refill rate | Applies to |
|---|---|---|---|
| `writes/sec` | 10 | 10 tokens/sec | any mutating tool call |
| `writes/min` | 60 | 60 tokens/60s | any mutating tool call (secondary gate) |
| `concurrent` | 5 | n/a (semaphore) | any tool call in flight |

Plus one per-request cap:

- `maxPayloadBytes`: 1,048,576 (1 MiB). Applies to the serialized tool-args JSON. Oversize request rejected immediately with `RATE_LIMITED` and `details: { reason: 'payload_too_large', limit, size }`.

Read-only tool calls (`maad_summary`, `maad_get`, `maad_query`, etc.) are exempt from the write buckets but still count against `concurrent` and `maxPayloadBytes`. Reads are cheap; the concurrent cap protects against pathological parallel-reads-from-one-session scenarios without penalizing normal behavior.

### Why two buckets for writes

The `writes/sec=10` bucket allows brief bursts (e.g. 10 rapid creates while setting up a brain). The `writes/min=60` bucket caps sustained rate. A client that writes 10 ops per second for 6 seconds straight will hit the minute cap. Two buckets are standard token-bucket shape; one cap alone can't express both "allow bursts" and "cap sustained."

### Bucket storage

Per-session buckets in-memory in the MCP server process, keyed by `sessionId`. When a session disconnects, its buckets are garbage-collected.

`src/mcp/rate-limit.ts` (new):

```ts
interface TokenBucket {
  capacity: number;
  refillPerSec: number;
  tokens: number;
  lastRefillMs: number;
}

class SessionRateLimiter {
  private writesSec = new Map<string, TokenBucket>();
  private writesMin = new Map<string, TokenBucket>();
  private concurrent = new Map<string, number>();  // in-flight count

  tryAcquireWrite(sessionId: string): { ok: true } | { ok: false; reason: string; retryAfterMs: number };
  tryAcquireConcurrent(sessionId: string): { ok: true; release: () => void } | { ok: false };
  checkPayloadSize(bytes: number): { ok: true } | { ok: false; limit: number };
  disposeSession(sessionId: string): void;
}
```

Token refill is lazy — computed on-demand when `tryAcquire*` is called, not via a background timer. Keeps the implementation free of timer management.

### Env knobs

| Var | Default | Meaning |
|---|---|---|
| `MAAD_RL_WRITES_PER_SEC` | 10 | writes/sec bucket capacity + refill rate |
| `MAAD_RL_WRITES_PER_MIN` | 60 | writes/min bucket capacity + refill rate |
| `MAAD_RL_CONCURRENT` | 5 | max concurrent in-flight requests per session |
| `MAAD_RL_PAYLOAD_BYTES` | 1048576 | max serialized args size |
| `MAAD_RL_DISABLE` | unset | if `"1"`, bypasses all rate-limit checks (for tests, benchmarking, debugging) |

All read at startup; no hot-reload. Restart to change.

### Rejection response

New error code:

```ts
| 'RATE_LIMITED'
```

Response body includes:

```json
{
  "ok": false,
  "errors": [{
    "code": "RATE_LIMITED",
    "message": "Rate limit exceeded",
    "details": {
      "reason": "writes_per_sec" | "writes_per_min" | "concurrent" | "payload_too_large",
      "limit": 10,
      "retryAfterMs": 450
    }
  }]
}
```

`retryAfterMs` is the earliest time (in milliseconds from now) the caller can retry. For the concurrent cap it's null — retry whenever an in-flight request completes. For payload-too-large it's null — resending won't help.

### Client retry contract

To prevent retry storms and self-inflicted DDOS, clients observing `RATE_LIMITED` **MUST** follow these rules:

1. **Honor `retryAfterMs`.** If present and non-null, do not retry before this much time has elapsed.
2. **Exponential backoff on repeated rejection.** On consecutive rejections, multiply wait time by 2 (or use full-jitter exponential backoff). Cap at 60 seconds.
3. **Give up after N consecutive rejections.** Recommended cap: 5 consecutive rejections, then surface the error to the human caller rather than spinning forever.
4. **Treat `payload_too_large` as permanent.** The payload will not shrink on retry. Do not retry without modifying the request.
5. **Treat `concurrent` as transient.** Retry is safe, but backoff still applies.

The server does not enforce these rules — it is always allowed to reject. Clients that ignore the contract will be rate-limited more aggressively, not less.

Idempotency keys (H4) are strongly recommended on any write that might be retried under rate limiting. Without a key, a retry that lands *after* the original eventually succeeded produces a duplicate record.

### Wiring

Rate limit check runs inside `withSession` / `withEngine` wrappers in `src/mcp/with-session.ts`, *after* session validation and *before* the engine call. Order of checks in the wrapper:

1. Session bound? → `SESSION_UNBOUND` if not
2. Role sufficient for this tool? → `INSUFFICIENT_ROLE` if not
3. Project whitelisted? → `PROJECT_NOT_WHITELISTED` if not
4. **Payload size check** → `RATE_LIMITED`
5. Idempotency cache check → replay if hit (skips remaining checks)
6. **Concurrent cap check** → `RATE_LIMITED`
7. **Write rate check (if mutating tool)** → `RATE_LIMITED`
8. Call engine method
9. Release concurrent slot (in `finally`)

Order matters: payload check first because oversize args waste compute. Idempotency before rate-limit because a replay should not consume tokens. Concurrent release in `finally` so an engine error doesn't leak a slot.

## Test plan

`test/mcp/rate-limit.test.ts`:

### R1 — Writes/sec burst
10 rapid `maad_create` calls all succeed. 11th within the same second gets `RATE_LIMITED` with `reason: writes_per_sec`. After 100ms wait, a fresh call succeeds (one token refilled).

### R2 — Writes/min cap
Launch 60 creates at 10/sec with small pauses. 61st fails with `reason: writes_per_min`. After 1s more elapsed, still fails (minute window not yet refilled). Advance mock clock 60s, succeeds.

### R3 — Concurrent cap
Start 5 long-running writes (use a slow-mock backend). 6th returns `RATE_LIMITED` with `reason: concurrent`. Release one, 6th succeeds on retry.

### R4 — Payload cap
Call with args serializing to >1 MiB. Rejected with `reason: payload_too_large`, no engine call made.

### R5 — Reads exempt from write bucket
Exceed writes/sec, then call `maad_summary` — succeeds. Proves reads don't consume write tokens.

### R6 — Session isolation
Two sessions, each hitting their own rate caps independently. Session A maxed out; session B still works.

### R7 — Disable env
Set `MAAD_RL_DISABLE=1`, launch 1000 creates — none rate-limited. Proves the kill switch works.

### R8 — Session disposal
Create session, exceed rate cap, disconnect, reconnect with same id policy — new session gets fresh buckets (garbage collection worked).

## Interactions

- **Mutex (H2):** rate limit fires before the mutex. A rejected request never queues.
- **Idempotency (H4):** replay hits skip the rate limit entirely. Fresh writes pay one token.
- **Logging (H6):** every rejection emits a log line with `sessionId`, `tool`, reason, and current bucket state. Useful for diagnosing a client that's getting throttled.
- **Graceful shutdown (H7):** during shutdown, new requests are rejected with `SHUTTING_DOWN` *before* the rate-limit check runs. Rate-limit state is in-memory and discarded at exit.
