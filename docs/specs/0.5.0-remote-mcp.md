# MAADB v0.5.0 — Remote MCP Transport (HTTP/SSE)

**Status:** spec
**Target release:** 0.5.0
**Depends on:** 0.4.0 (multi-project routing), 0.4.1 (production hardening)
**Enables:** 0.5.1 (deployment workflow), 0.6.0 (npm package), 0.8.5 (remote hardening)

## Goal

Serve MAADB over HTTP/SSE so MCP clients can connect across the network. One server process, many concurrent client sessions, bearer-token auth at handshake, concurrent reads, polling delta tool. stdio remains the default transport for local use.

## Non-goals

- Per-connection role downgrades (deferred to 0.8.5; whole-session role from token tier)
- Token rotation without restart (deferred to 0.8.5)
- Webhooks / push notifications (`maad_changes_since` is poll-based)
- TLS termination in-process (assumed at upstream reverse proxy)
- Multi-tenant isolation across instances (one process = one instance, same as 0.4.0)
- Web UI / browser client (CLI/MCP clients only)
- Cross-project queries (each call still hits exactly one project, per 0.4.0)

## Architecture

```
┌─ Reverse proxy (TLS, optional) ─────────────────┐
│  https://brain.example.com → http://127.0.0.1   │
└──────────────────┬──────────────────────────────┘
                   │
┌─ MCP Server process (transport=http) ───────────┐
│                                                  │
│  HTTP listener (StreamableHTTPServerTransport)   │
│    ├─ POST /mcp        client → server (JSON)    │
│    ├─ GET  /mcp        SSE stream server → client│
│    └─ DELETE /mcp      explicit session close    │
│                                                  │
│  authMiddleware  → bearer token validation       │
│         │                                         │
│         ▼                                         │
│  Instance (unchanged from 0.4.0)                 │
│    ├─ EnginePool                                 │
│    └─ SessionRegistry                            │
│         ├─ onclose(sessionId) fan-out:           │
│         │    rateLimiter.drop(sessionId)         │
│         └─ idle timeout sweeper                  │
└──────────────────────────────────────────────────┘
```

## Configuration

### CLI flags / env vars

| Flag | Env var | Default | Purpose |
|------|---------|---------|---------|
| `--transport <stdio\|http>` | `MAAD_TRANSPORT` | `stdio` | Transport selection |
| `--http-host <host>` | `MAAD_HTTP_HOST` | `127.0.0.1` | Bind address (loopback default — proxy-fronted) |
| `--http-port <port>` | `MAAD_HTTP_PORT` | `7733` | Bind port |
| `--auth-token <token>` | `MAAD_AUTH_TOKEN` | *required for http* | Bearer token (opaque, ≥32 bytes recommended) |
| `--session-idle-ms <ms>` | `MAAD_SESSION_IDLE_MS` | `1800000` (30 min) | Idle session eviction threshold |
| `--http-max-body <bytes>` | `MAAD_HTTP_MAX_BODY` | `1048576` (1 MiB) | Max request body (matches existing payload cap) |
| `--trust-proxy` | `MAAD_TRUST_PROXY` | `false` | Trust `X-Forwarded-For` first hop for client IP in logs |
| `--http-headers-timeout <ms>` | `MAAD_HTTP_HEADERS_TIMEOUT_MS` | `10000` (10s) | `node:http` `headersTimeout` — slowloris defense |
| `--http-request-timeout <ms>` | `MAAD_HTTP_REQUEST_TIMEOUT_MS` | `60000` (60s) | `node:http` `requestTimeout` — hung inbound request defense |
| `--http-keepalive-timeout <ms>` | `MAAD_HTTP_KEEPALIVE_TIMEOUT_MS` | `5000` (5s) | `node:http` `keepAliveTimeout` — connection reuse window |

**Validation at boot:**
- `transport=http` without `MAAD_AUTH_TOKEN` → fail fast with `AUTH_TOKEN_REQUIRED`
- Token shorter than 16 chars → warn (don't fail; allow dev convenience)
- Bind to non-loopback host without TLS hint → log warning recommending reverse proxy
- `MAAD_TRUST_PROXY=true` only safe behind a proxy you control that strips/sets `X-Forwarded-For`. Untrusted upstream = log spoofing surface. Default `false` falls back to socket `remoteAddress`

**Timeout defaults matter.** `node:http`'s stock defaults are not production-safe (slowloris, hung-request, unbounded keepalive). Framework users get these set implicitly by Express/Fastify; we set them explicitly against the raw `node:http` server. The defaults above are recommended production values — tune for your environment.

**Where the timeouts are applied.** The SDK's `StreamableHTTPServerTransport` is a pure request handler — it does NOT own the HTTP server or its timeouts. Our `src/mcp/transport/http.ts` creates the `node:http` server, sets `server.headersTimeout` / `server.requestTimeout` / `server.keepAliveTimeout` from the config values above, wires the request listener through auth middleware into `transport.handleRequest(req, res)`, then calls `server.listen(host, port)`. The SDK's Node.js path may internally use `@hono/node-server`; if so, we configure timeouts on the underlying server instance, not the hono wrapper. (R1 smoke test verifies timeouts actually fire.)

### Token shape

- Opaque random bytes, base64url-encoded (≥32 bytes / 43 chars after encoding recommended)
- Not a JWT — no claims, no expiration, no signing (single role tier per roadmap)
- Comparison via constant-time equality (`crypto.timingSafeEqual`)
- One token per server process; multi-token / per-client tokens deferred to 0.8.5

## Auth flow

```
Client                          Server
  │                                │
  │  POST /mcp (initialize)        │
  │  Authorization: Bearer <tok>   │
  │ ──────────────────────────────►│
  │                                │  authMiddleware:
  │                                │    extract bearer
  │                                │    timingSafeEqual vs MAAD_AUTH_TOKEN
  │                                │    miss → 401 {code: UNAUTHORIZED}
  │                                │    hit  → attach { authenticated: true } to req
  │                                │
  │                                │  StreamableHTTPServerTransport:
  │                                │    create session, assign sessionId
  │                                │    wire to SessionRegistry
  │  ◄──────────────────────────── │  200 + mcp-session-id header
  │                                │
  │  GET /mcp (SSE)                │
  │  mcp-session-id: <id>          │
  │  Authorization: Bearer <tok>   │
  │ ──────────────────────────────►│  authMiddleware re-validates
  │  ◄═══════════ SSE stream ═════ │
```

**Every request re-validates the token.** No "trust the session ID alone" shortcut — session hijack via guessed/leaked ID alone gets nothing without the token.

### Middleware order (strict)

```
1. authMiddleware        → bearer token check
                            miss/invalid → 401 UNAUTHORIZED, stop
2. sessionResolution     → mcp-session-id lookup (if present)
                            unknown id → 404 SESSION_NOT_FOUND
3. rateLimit / shutdown  → existing 0.4.1 gates
4. handler               → tool dispatch
```

Auth runs **before** session resolution. Unauthenticated callers can never reach a 404, so the existence of any session ID is never disclosed to them. 404 only ever returned to authenticated callers.

**401 response shape:**
```json
{ "ok": false, "errors": [{ "code": "UNAUTHORIZED", "message": "missing or invalid bearer token" }] }
```

No detail leak (don't say *which* check failed, don't echo the presented token).

### Response hardening (all HTTP responses)

Minimum header set applied by the HTTP layer:

| Header | Value | Purpose | Who sets it |
|---|---|---|---|
| `Cache-Control` | `no-store` on POST/DELETE JSON responses | Prevent intermediaries from caching auth/tool responses | **Us** (inject via wrapper — SDK emits only `no-cache` / `no-cache, no-transform`) |
| `Cache-Control` | `no-cache, no-transform` on SSE GET | Keep SSE streams uncached and unmodified | **SDK** (already emits) |
| `X-Content-Type-Options` | `nosniff` | Block MIME sniffing on error bodies | **Us** (SDK does not emit) |
| `Content-Type` | `application/json` (POST/DELETE) / `text/event-stream` (SSE GET) | Explicit, no guessing | **SDK** (already emits) |

Cheap hygiene; no Helmet dep needed. Implemented in the HTTP transport wrapper layered above `handleRequest` — we post-process the outgoing response (or intercept the `Response` object in the Web Standard path) to inject the two "us"-owned headers before the response hits the wire.

## Session lifecycle

### Open
1. `initialize` request authenticated + accepted by transport
2. **Session ID generation is delegated to us by the SDK.** The SDK transport does not generate IDs itself — it calls a `sessionIdGenerator` callback we supply at transport construction. We supply `() => crypto.randomBytes(16).toString('base64url')` — 128 bits of CSPRNG entropy (guaranteed, vs. `crypto.randomUUID()`'s ~122 bits). SDK rejects client-supplied session IDs on initialize and returns 404 for any inbound `mcp-session-id` it doesn't know, confirming our "server-generated only" invariant.
3. Transport fires `onsessioninitialized(sessionId)` → `SessionRegistry.create(sessionId)`; `SessionState` with `createdAt`, `lastActivityAt = now`
4. Audit log: `session_open { sessionId, remoteAddr, userAgent }`

### Active
- **`lastActivityAt` updates only on inbound client requests** (POST/GET/DELETE on `/mcp`)
- Outbound SSE pushes from server do NOT count as activity — server activity ≠ client activity
- Result: open-but-quiet SSE connections still get evicted at idle threshold; prevents zombie streams from sitting open forever
- `withSession` routes per-call (project binding, role check, mutex/no-mutex)
- Rate-limit token bucket scoped per `sessionId` (existing 0.4.1 behavior)

### Close (any cause)
Triggered by:
- Client `DELETE /mcp` (graceful)
- Transport `onclose` (TCP drop, SSE break)
- Idle sweeper (`now - lastActivityAt > MAAD_SESSION_IDLE_MS`)
- Server shutdown (drain phase closes all sessions)

**Fan-out (sequential, in order):**
1. `SessionRegistry.delete(sessionId)` — removes binding state
2. `rateLimiter.drop(sessionId)` — releases token buckets, frees concurrency slot
3. Audit log: `session_close { sessionId, reason, durationMs }`

**Idempotency cache is intentionally NOT in the fan-out.** 0.4.1 idempotency is keyed `(project, tool, key)` with no session component, so there is no per-session state to evict. The 10-minute TTL LRU handles its own cleanup. A reconnecting client with the same idempotency key correctly hits the replay path within the TTL window — desirable behavior.

This closes the deferred polish-pass item (session disconnect cleanup) — scoped down to what actually has session state to clean.

### Idle sweeper
- Single `setInterval` on the registry, tick every 60s, `unref()` so it doesn't block exit
- Walks `SessionRegistry`, closes any session past idle threshold
- Counts/timestamps surfaced in `maad_health`

## Concurrent reads

**Current state (0.4.1):** `runExclusive(mutex, fn)` wraps every mutating op. Reads currently route through `withEngine` but don't take the mutex (read path is pure file reads + SQLite query). Confirmed safe.

**0.5.0 change:** make this explicit and tested.

- Add `OperationKind = 'read' | 'write'` to tool registration metadata
- `withSession` checks kind:
  - `read` → invoke handler directly, no mutex
  - `write` → wrap in `runExclusive(engine.writeMutex, handler)`
- Reads remain unblocked while a write is in flight
- Multiple concurrent reads execute in parallel (limited only by SQLite's connection pool)

### Hard rule (enforced by review, not yet linter)

> **A `read` tool MUST NOT mutate git state, SQLite state, in-memory cache state, or filesystem state.** Every tool registration MUST declare `OperationKind` explicitly — no defaults, no naming heuristics, no inference from prefix. PR review checklist line: "every modified/added tool declares `OperationKind` and a misclassified read would corrupt data."

Lint enforcement deferred (would need AST analysis of each handler's call graph). Logged as followup.

**New test:** `tests/transport/concurrent-reads.test.ts` — N concurrent reads complete while a slow write holds the mutex; assert wall-clock < N × read latency.

## `maad_changes_since`

New read tool. Polling delta — clients poll for "what changed since I last asked".

### Schema

```ts
{
  project: string,           // multi-project mode
  cursor?: string,           // opaque cursor from prior response; omit on first call
  limit?: number,            // default 100, max 1000
  docTypes?: string[],       // optional filter
}
```

### Response

```json
{
  "ok": true,
  "data": {
    "changes": [
      { "docId": "sess-...", "docType": "session", "updatedAt": "2026-04-15T14:23:01.234Z", "operation": "update" }
    ],
    "nextCursor": "eyJ1IjoiMjAyNi0wNC0xNVQxNDoyMzowMS4yMzRaIiwiZCI6InNlc3MtYWJjIn0",
    "hasMore": false
  }
}
```

### Cursor semantics

- Sort order is **strict and deterministic**: `ORDER BY updated_at ASC, doc_id ASC`
- Cursor is an opaque base64url-encoded blob of `{u: updatedAt, d: docId}` — clients treat it as a token, never parse it
- Resume rule: `(updated_at, doc_id) > (cursor.u, cursor.d)` — strict tuple comparison, no inclusive/exclusive ambiguity
- Timestamp ties resolved by `doc_id` ordering — no possibility of duplicate emission across pages or polls
- First call: omit `cursor` to start from the beginning (full snapshot, paginated via `hasMore`)
- `nextCursor` always returned when `hasMore=true`; equals the `(updated_at, doc_id)` of the last returned change
- Server clock is authoritative; clients pass `nextCursor` back verbatim
- Cursor format may evolve in future versions; opaqueness is the contract that lets us change it

### `operation` field

- `create` — first appearance in index
- `update` — `updated_at > created_at`
- `delete` — placeholder; deletion tracking deferred (current engine doesn't tombstone)

For 0.5.0, delete events are NOT emitted. Documented limitation. Followup logged for 0.7.0+ when `_inbox/` import work touches the doc lifecycle.

### Index requirement

Existing `documents.updated_at` (added in 0.2.8) is sufficient. Add covering index `(updated_at, doc_type)` if query plan shows scan — measure first.

## Health surface additions

`maad_health` extends with:

```json
{
  "transport": {
    "kind": "http",
    "host": "127.0.0.1",
    "port": 7733,
    "uptimeSeconds": 12345
  },
  "sessions": {
    "active": 4,
    "openedTotal": 127,
    "closedTotal": 123,
    "lastOpenedAt": "2026-04-15T14:20:00.000Z",
    "lastClosedAt": "2026-04-15T14:22:31.000Z",
    "idleSweepLastRunAt": "2026-04-15T14:23:00.000Z"
  }
}
```

Under `transport=stdio`, the `transport` block reports `kind=stdio` and omits host/port; `sessions` always has at most 1 active.

**Auth requirement:** under `transport=http`, `maad_health` (the rich tool) requires the bearer token like every other tool. Health output reveals project names, doc counts, and transport posture — recon-useful data. Under `transport=stdio` the tool remains unauthenticated (local trust).

### `GET /healthz` — unauthenticated liveness probe

Container runtimes and orchestrators (systemd watchdog, Docker HEALTHCHECK, Kubernetes/ACA liveness probes) need a cheap, unauthenticated endpoint to answer "is this process alive?". Requiring the bearer token for liveness would force the orchestrator to hold a copy of the secret, which creates secret-distribution overhead at the hosting layer for zero security benefit — the orchestrator already controls the process.

`GET /healthz` returns a minimal liveness response:

```json
{ "ok": true }
```

Rules:
- Unauthenticated (no bearer required)
- HTTP 200 when the process is up and the listener is responding
- HTTP 503 `SHUTTING_DOWN` during drain (matches the single-code shutdown contract)
- No internal state in the body — no project names, no doc counts, no transport details, no session counts. Nothing a scanner can use for reconnaissance
- Not exposed under `transport=stdio`

Operators who need richer ops data call the authenticated `maad_health` tool. Liveness ≠ health; this split matches the standard k8s `livenessProbe` vs `readinessProbe` distinction.

## New error codes

| Code | When | HTTP status |
|------|------|-------------|
| `AUTH_TOKEN_REQUIRED` | Boot: `transport=http` without `MAAD_AUTH_TOKEN` | n/a (boot fail) |
| `UNAUTHORIZED` | Request without valid bearer token | 401 |
| `SESSION_NOT_FOUND` | Authenticated request with unknown `mcp-session-id` | 404 |
| `IDLE_TIMEOUT` | Session evicted by sweeper (close-audit reason, not a request response) | n/a |

`SHUTTING_DOWN` from 0.4.1 is the **single** drain/shutdown response. Every request that arrives during draining or after listener close → 503 `SHUTTING_DOWN`. No second code for the post-listener-close case — clients see one consistent signal regardless of where in the shutdown lifecycle the request landed.

## R0 findings (completed 2026-04-15)

SDK inspection at `@modelcontextprotocol/sdk@1.29.0`. Full findings captured in `jrn-2026-007`.

**Resolved by R0:**

| Question | Answer |
|---|---|
| 0. SDK convention review | Done. Drift items folded into spec above. |
| 1. SDK version compat | 1.29.0 supports `StreamableHTTPServerTransport`, `onclose`, `onsessioninitialized`, `onsessionclosed`. No bump needed. |
| 2. Body parsing / framework | `node:http` + SDK transport, no Express. SDK is auth-agnostic and path-agnostic; we own routing and middleware. |
| 3. CORS | Skip — no browser clients. Revisit if that changes. |
| 4. `/mcp` path | SDK has no hard-coded path. Examples use `/mcp` for POST/GET/DELETE. We follow convention. |

**Drift folded into spec above:**
- Header name is lowercase `mcp-session-id` (SDK normalization)
- Session IDs come from our `sessionIdGenerator` callback — SDK does not generate. We supply `crypto.randomBytes(16).toString('base64url')` (128 bits guaranteed vs. `randomUUID`'s ~122)
- `Cache-Control: no-store` and `X-Content-Type-Options: nosniff` are ours to inject; SDK emits only `no-cache` variants
- `node:http` timeouts applied on the raw server we create, not on the SDK transport

**Surviving open questions (R1-deferred, answerable only by writing code):**

1. **Response wrapping mechanics** — inject hardening headers via wrapper around `handleRequest`, or post-process the `Response` object in the Web Standard path? Decide during R1 once we see which path the SDK's Node.js wrapper actually takes.
2. **Timeout propagation through `@hono/node-server`** — if the SDK's Node.js wrapper internalizes hono, verify our timeouts on the underlying server instance actually fire. R1 smoke test.

**Closed as decisions:**

3. **Auth token format validation at parse time** — accept any string, no base64url-format check. Only constant-time equality matters. Length-warning at boot stays.
4. **Transport ↔ session mapping** — 1:1. One transport instance per session, stored in a `transports[sessionId]` map. No multiplexing.
5. **Read concurrency cap per session** — trust existing 0.4.1 rate limiter (10/sec, 5 concurrent per session). Add a cap only if R1/R4 smoke tests show SQLite saturation under adversarial flood.

R0 complete. R1 is unblocked.

## Risks

- **SDK churn** — `StreamableHTTPServerTransport` is newer than stdio transport in MCP SDK; API may shift. Mitigation: pin SDK version, write thin transport adapter so a swap is local
- **Auth token leak** — biggest single risk. Mitigations: redaction, no token in error messages, document rotation = restart, recommend short token lifetime + manager-driven rotation in deploy guide
- **Read/write split bug** — accidentally annotating a write as a read corrupts data. Mitigation: explicit `OperationKind` per tool, lint rule TODO (deferred), code review checklist line
- **SSE behind proxies** — some proxies buffer SSE, breaking streaming. Mitigation: deploy guides set `proxy_buffering off` (nginx) and equivalent
