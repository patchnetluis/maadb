# 0.4.1 — Idempotency Keys on Writes

Spec for optional client-supplied idempotency keys that deduplicate retried writes within a TTL window. Prevents the "retry on timeout creates a duplicate record" bug.

## Problem

With the 0.4.1 write mutex blocking indefinitely and network transports (0.5.0 HTTP/SSE arriving next) introducing new retry paths, a client that retries after a perceived timeout can land the same logical write twice. MAADB assigns a new docId on each request unless the caller specifies one, so two retries of a "create agent named Acme" produce two Acme records. Silent corruption of the brain.

The existing `expectedVersion` mechanism on `updateDocument` protects against *concurrent* writes overwriting each other, but does nothing for *retries of the same write*.

## Goals

- Client can opt into exactly-once semantics for a write by supplying an idempotency key.
- Second occurrence of the same key within the TTL returns the *original* result, not a new one — including the same `docId`, `version`, and `validation` payload.
- No behavior change for clients that omit the key.
- Storage is bounded — old keys expire automatically.

## Non-goals (deferred / out of scope)

- **Automatic key generation on the server.** Key must be client-supplied; the server does not hash request bodies. Body-hash idempotency is a different contract and belongs in a later version if demand justifies.
- **Cross-engine key sharing.** Keys are per-engine (per project). A key reused across two projects is two independent keys.
- **Persistent key store.** Keys live in-memory. A restart clears them. Acceptable because retry windows are short and clients hold their own copies.

## Design

### Where the check lives

At the **MCP tool layer**, not inside the engine. Reasoning:

- The engine's mutating methods already assume the caller's request is logically distinct. Pushing idempotency into `writes.ts` entangles the write pipeline with request-protocol concerns.
- MCP tool handlers are the natural boundary: they own the request envelope, which is where the idempotency key rides.
- If a future CLI or programmatic caller wants idempotency, they can layer their own replay cache — the MCP layer's logic is straightforward to copy.

Implementation location: `src/mcp/idempotency.ts` (new) plus thin hooks in each mutating tool handler (`src/mcp/tools/write.ts`).

### Key shape

Client sends an `idempotencyKey` field in the tool arguments — a free-form string ≤128 chars, whatever UUID/hash scheme they prefer. Server treats it as an opaque bytes-of-text identifier.

Scope: **per project, per tool**. Tuple `(projectName, toolName, idempotencyKey)` is the cache key. Same key across different tools is not a collision; keys are shared across all sessions bound to the same project.

**Why project-scoped, not session-scoped.** The main point of idempotency is retry survival. If a client's connection drops mid-retry and it reconnects with a fresh session ID, a session-scoped cache would miss and execute the write twice. Project-scoped cache replays cleanly regardless of session lifecycle.

**Collision concern.** Two unrelated clients in the same project could theoretically use the same key string. Clients solve this by using UUIDs or sufficiently-random identifiers — it is the client's responsibility to produce keys that do not collide, just as with idempotency implementations in Stripe, AWS, etc. The server does not police the key space.

### Cache

In-memory LRU with TTL, `src/mcp/idempotency.ts`:

```ts
interface IdempotencyEntry {
  key: string;                    // (projectName, toolName, idempotencyKey) composed
  response: unknown;              // the original CallToolResult (with _meta stripped — replay rebuilds _meta fresh)
  originalRequestId: string;      // request_id from the first execution
  originalCompletedAt: string;    // ISO timestamp of first execution completion
  createdAt: number;              // ms epoch (for TTL + LRU)
}

class IdempotencyCache {
  private ttlMs = 10 * 60 * 1000;    // 10 minutes
  private maxEntries = 10_000;
  private entries = new Map<string, IdempotencyEntry>();

  get(key: string): IdempotencyEntry | null { /* + TTL check + LRU reorder */ }
  put(key: string, response: unknown): void { /* + evict oldest if over cap */ }
}
```

Single instance per MCP server process, shared across all sessions and projects (the composed key includes `projectName` so no cross-project leak).

TTL: **10 minutes**. Covers realistic retry windows (client timeout + backoff + retry) without holding write responses in memory longer than useful. Tunable via env `MAAD_IDEMPOTENCY_TTL_MS`.

Max entries: **10,000**. Simple LRU eviction when full. Tunable via env `MAAD_IDEMPOTENCY_MAX`.

### Tool handler wiring

Every mutating tool (`maad_create`, `maad_update`, `maad_delete`, `maad_bulk_create`, `maad_bulk_update`) accepts an optional `idempotencyKey` in its input schema. The wrapper does:

```ts
// pseudocode inside write tool handlers
async function handle(args, extra) {
  const cacheKey = args.idempotencyKey
    ? composeKey(activeProject, 'maad_create', args.idempotencyKey)
    : null;

  if (cacheKey) {
    const cached = idempotencyCache.get(cacheKey);
    if (cached) {
      log('idempotency_replay', { tool, project: activeProject, key: args.idempotencyKey });
      // Return original response body, but attach fresh _meta identifying the replay
      return {
        ...cached.response,
        _meta: {
          request_id: currentRequestId,
          replayed: true,
          original_request_id: cached.originalRequestId,
          original_completed_at: cached.originalCompletedAt,
        },
      };
    }
  }

  const response = await /* normal write path */;

  if (cacheKey && responseWasSuccessful(response)) {
    idempotencyCache.put(cacheKey, {
      response,
      originalRequestId: currentRequestId,
      originalCompletedAt: new Date().toISOString(),
    });
  }
  return response;
}
```

Three deliberate choices:

1. **Only successful responses are cached.** A failed write should not be treated as "done" on retry — the client might be retrying precisely because the first attempt failed. Failed responses do not go into the cache; the retry runs the write for real.
2. **Replay is identifiable, not transparent.** The response body (`ok`, `data`, `errors`) is byte-identical to the original. But `_meta.replayed: true` plus `_meta.original_request_id` and `_meta.original_completed_at` let clients detect a replay if they care (telemetry, debugging). Clients that ignore `_meta` see the same response shape as a fresh successful call. This replaces the earlier "fully transparent" design — identifiability costs nothing and makes post-mortems dramatically easier.
3. **No mutex interaction.** The cache check happens *before* acquiring the mutex. A replay hit skips the mutex entirely and returns instantly. Fresh writes proceed to the mutex as normal.

### Error code

```ts
| 'IDEMPOTENCY_REPLAY'   // emitted only in logs/audit channel, not in response
```

Not used as a visible error code to the client — the whole point is that replay is transparent. The code exists for log emission and potential future introspection tools.

## Test plan

`test/mcp/idempotency.test.ts`:

### I1 — Same key returns cached response
Two calls to `maad_create` with identical args + same `idempotencyKey`. Second resolves to the same `docId`, `version`, and `filePath` as the first. Backend has one document, not two.

### I2 — Different keys produce different writes
Two calls to `maad_create` with identical args but different keys. Both succeed, both produce distinct docIds, backend has two documents.

### I3 — No key = no dedup
Two calls to `maad_create` with identical args, no `idempotencyKey` on either. Produces two distinct documents (existing behavior preserved).

### I4 — Failed write is not cached
Stage a `maad_create` with invalid fields (validation will fail) and an idempotencyKey. Second call with same key + valid fields: succeeds and creates the record. The failure was not cached.

### I5 — TTL expires
Put an entry, advance a mocked clock past 10 minutes, same key hits: cache miss, new write executes. Test uses a injectable clock in the `IdempotencyCache`.

### I6 — Project scoping + session independence
- Two sessions bound to the **same project** with the same `idempotencyKey` → second call replays (proves session lifecycle does not affect cache key).
- Two sessions bound to **different projects** with the same `idempotencyKey` string → both writes execute distinctly (proves project isolation).

### I6b — Replay identifiability
Replay response has:
- identical `ok` / `data` / `errors` fields
- `_meta.replayed === true`
- `_meta.original_request_id` matches the `request_id` from the original call's ops log line
- `_meta.original_completed_at` is a valid ISO timestamp earlier than the replay timestamp

### I7 — LRU eviction
Fill cache to `maxEntries`, put one more, oldest entry evicted (cache miss on lookup). Test uses small cap (say 5) for tractability.

## Interactions

- **Mutex (H2):** cache check is before mutex acquisition. A replay hit skips the mutex entirely.
- **Rate limit (H5):** rate limiter fires *after* the idempotency cache check. A replay hit does not consume a rate-limit token. Rationale: replay is a logical no-op from the engine's perspective.
- **Logging (H6):** replay events emit a dedicated `idempotency_replay` log line with `sessionId`, `tool`, and key fingerprint (first 8 chars of key, not the full key) for debuggability.
- **Graceful shutdown (H7):** cache is in-memory; shutdown drops it. Clients that retry after a restart do not get replay — they run the write fresh. Documented behavior.
