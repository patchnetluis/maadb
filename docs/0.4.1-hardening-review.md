# MAADB 0.4.1 — Production Hardening (Review Bundle)

**Version:** MAADB 0.4.1 (planned)
**Base:** MAADB 0.4.0 (shipped 2026-04-14, tag `v0.4.0`, commit `86a9904`)
**Prepared:** 2026-04-14
**Status:** Specs accepted. Implementation not started.

This is a self-contained review bundle. It inlines the five component specs that make up the 0.4.1 release, plus the rationale for resequencing the public roadmap to put hardening before the remote-transport work in 0.5.0. It is suitable for sharing with reviewers who do not have access to the MAADB repository.

---

## Context

MAADB is a single-process engine that stores markdown records under a project directory, indexes them into SQLite, and auto-commits every change to git. The 0.4.0 release added multi-project routing and session-scoped mode to a single MCP server — multiple MAAD projects can now be served by one process, with each session binding to a project (or a whitelist). That unlocked the architectural runway for remote MCP transport, which lands next as 0.5.0.

Before the engine goes over a network, the write path and the operational surface need to be hardened. Under stdio transport today, the engine has been safe-by-accident: one Claude Code client drives one engine synchronously, with no concurrency and no retry loops. Once HTTP/SSE lands, N sessions can multiplex onto one engine, retries become commonplace, and a runaway client must not be able to saturate or corrupt the engine.

0.4.1 is that hardening pass. It introduces no new user-facing tools. It adds internal structure — a write mutex, idempotency keys, rate limiting, structured logs with an audit channel, per-request timeouts, graceful shutdown, extended health reporting — so that 0.5.0 can expose the engine over HTTP without obvious failure modes.

The shape changed from the prior sequence in three ways:

1. **0.4.1 inserted.** Previously the roadmap went straight from 0.4.0 multi-project routing into feature work (import, query power). Network exposure arriving in 0.5.0 means the hardening pass is the most important work on the board.
2. **0.5.0 Remote MCP pulled forward from 0.9.0.** It is the transport that makes hosted deployment possible. The original 0.9.0 slot assumed a later-and-larger scope; 0.5.0 ships it deliberately minimal (single role tier, polling delta instead of push subscriptions) on top of the hardened engine from 0.4.1.
3. **0.6.0 npm package pulled forward from 0.8.0.** Once the engine can run hosted, having it trivially installable into container images (`npx maad serve`) becomes load-bearing rather than cosmetic.

Everything deferred out of 0.4.1 and 0.5.0 lands in **0.8.5 Remote MCP hardening**, once real usage signal justifies the additional complexity.

## Canonical request flow (0.4.1 end-state)

Every MCP tool call, once the 0.4.1 items ship, runs through this ordered sequence:

```
1.  Session bound?                 → SESSION_UNBOUND if not
2.  Role sufficient for tool?      → INSUFFICIENT_ROLE if not
3.  Shutting down?                 → SHUTTING_DOWN if yes          (H7)
4.  Project whitelisted?           → PROJECT_NOT_WHITELISTED
5.  Payload size OK?               → RATE_LIMITED (payload)         (H5)
6.  Idempotency cache hit?         → return cached response         (H4)
7.  Concurrent slot available?     → RATE_LIMITED (concurrent)      (H5)
8.  Write rate budget available?   → RATE_LIMITED (rate)            (H5)
9.  Acquire engine write mutex     (blocks until FIFO turn)         (H2)
10. Execute engine method          (under per-request timeout)      (H7)
11. Release mutex                                                   (H2)
12. Cache response if successful                                    (H4)
13. Release concurrent slot                                         (H5)
14. Emit ops log line with request_id / latency / result            (H6)
15. If mutating and successful, emit audit log line                 (H6)
```

Reads skip steps 8, 9, 11, 12 (no mutex, no idempotency cache, no write budget). Everything else applies.

Each numbered step is owned by exactly one spec in this bundle. The interaction matrix at the end of each spec documents how items share state.

## Spec 1 of 5 — Write Concurrency

Spec for serializing mutating operations per project engine. Companion items in 0.4.1 (rate limiting, idempotency keys, logging, graceful shutdown) have their own specs; this one focuses solely on the write mutex, its error semantics, stale lock recovery, and the tests that prove it works.

### Problem

Today `MaadEngine` exposes mutating methods (`createDocument`, `updateDocument`, `deleteDocument`, `bulkCreate`, `bulkUpdate`, `indexAll`, `indexFile`, `reindex`) with no serialization. Under stdio transport that has been safe-by-accident: a single Claude Code client drives one engine synchronously. With HTTP/SSE transport arriving in 0.5.0, N connected sessions can multiplex onto one engine, and two concurrent writes against the same engine corrupt state along three failure modes:

1. **Git index race.** `git add` + `git commit` sequences in `src/git/commit.ts` are not mutex-protected. Two concurrent commits produce `fatal: Unable to create .git/index.lock: File exists` or, worse, a commit that silently stages only half a write's files.
2. **SQLite backend reindex.** `indexFile` issues multiple statements against the backend. SQLite WAL tolerates concurrent readers but serializes writers natively; two concurrent write transactions from the engine's perspective produce `SQLITE_BUSY` only after retry exhaustion, surfacing as an opaque `BACKEND_ERROR`.
3. **Journal entry interleaving.** `src/engine/journal.ts` keyed each operation by a monotonic ID, but two operations advancing stages concurrently can produce out-of-order journal files if the filesystem interleaves the writes at odd boundaries.

Single-writer constraint exists today at the SQLite layer by default, and at the git layer by OS file-lock collision. Neither produces clean, fair, predictable behavior under contention. This spec formalizes it.

### Goals

- **Correctness under contention.** Two concurrent writes to the same engine serialize cleanly; neither corrupts state.
- **Fair ordering.** First caller into the mutex is first served. FIFO.
- **Cheap.** No new processes, no new subprocess management. Mutex in-memory per engine instance.
- **Debuggable.** If a write stalls, the stall shows up in health output and logs within seconds.
- **Non-breaking.** Existing callers that never contend see no behavior change beyond a structured log line.

### Non-goals (captured in docs/gaps.md and deferred to 0.8.5)

- **Configurable acquire timeout returning `WRITE_TIMEOUT`.** 0.4.1 ships infinite block. Adding a timeout ratifies a policy question (how long to wait?) before evidence justifies a specific answer.
- **Queue-depth backpressure / 429-equivalent.** The 0.4.1 rate limiter fires at the session boundary; mutex contention is invisible to it. 0.8.5 adds a queue-depth health signal and distinct error code.
- **Cross-process coordination.** One engine instance owns one project. Two processes pointing at the same project root is unsupported configuration; no in-engine work protects against it. Operators who need HA run one instance and proxy to it.
- **Stress / throughput benchmarks.** 0.4.1 ships correctness smoke tests. 0.8.5 adds sustained-load suite.

### Design

#### Mutex shape

One **async FIFO mutex** per `MaadEngine` instance. Every public method that mutates engine-owned state (filesystem writes under `projectRoot`, SQLite backend, git commits, journal) acquires the mutex on entry and releases on exit — including error paths.

Target signature on `MaadEngine`:

```ts
private writeLock: AsyncFifoMutex = new AsyncFifoMutex();

private async runExclusive<T>(op: string, fn: () => Promise<T>): Promise<T> {
  const release = await this.writeLock.acquire();
  this.lastWriteOp = { op, startedAt: Date.now() };
  try {
    return await fn();
  } finally {
    this.lastWriteOp = null;
    release();
  }
}
```

Every mutating public method wraps its body:

```ts
async createDocument(...) {
  if (this._readOnly) return singleErr('READ_ONLY', '...');
  return this.runExclusive('createDocument',
    () => writes.createDocument(this.ctx(), ...),
  );
}
```

Methods to wrap (complete list):

| Method | Why |
|---|---|
| `createDocument` | file write + index + git commit |
| `updateDocument` | file write + index + git commit |
| `deleteDocument` | file rename/unlink + index + git commit |
| `bulkCreate` | N file writes + N indexes + 1 git commit |
| `bulkUpdate` | N file writes + N indexes + 1 git commit |
| `indexAll` | backend batch write |
| `indexFile` | backend write (called inside writes — internal caller bypass needed, see below) |
| `reindex` | backend rebuild + optional git operations |
| `reload` | backend tear-down + re-init (see Open Questions) |

#### Internal vs external entry points

`indexFile` is called both as a public method *and* internally from `createDocument` / `updateDocument` / `bulkCreate`. If the internal callers also acquire the mutex they deadlock.

Resolution: expose `indexFile` at the public API, but route the public entry through `runExclusive` and have the domain functions in `writes.ts` call the non-guarded `indexing.indexFile(ctx, ...)` directly. This is how the code is already structured — the split is free.

Same pattern for any future "internal helper that happens to be public."

#### Mutex implementation

`AsyncFifoMutex` — a small in-module class, no new dependency. Shape:

```ts
class AsyncFifoMutex {
  private queue: Array<() => void> = [];
  private held = false;

  acquire(): Promise<() => void> {
    return new Promise(resolve => {
      const grant = () => {
        this.held = true;
        resolve(() => this.release());
      };
      if (!this.held) grant();
      else this.queue.push(grant);
    });
  }

  private release(): void {
    this.held = false;
    const next = this.queue.shift();
    if (next) next();
  }

  depth(): number {
    return this.queue.length + (this.held ? 1 : 0);
  }
}
```

FIFO because a single `queue` array with `push` + `shift` is first-in-first-out. No priority inversion, no starvation, no external dependency.

#### Health integration

Extend `HealthReport`:

```ts
interface HealthReport {
  // ...existing fields...
  writeQueueDepth: number;        // writeLock.depth()
  lastWriteOp: {                  // null if no write currently in flight
    op: string;
    startedAt: string;            // ISO timestamp
    elapsedMs: number;
  } | null;
}
```

Emits into the structured log on every completion:

```json
{
  "event": "write_complete",
  "op": "createDocument",
  "elapsed_ms": 87,
  "queue_depth_on_enter": 2,
  "queue_depth_on_exit": 1
}
```

Long-running writes (>500ms) emit a `write_slow` warning with the op name and elapsed. Threshold is a module constant, not a config knob, for 0.4.1.

### Stale `.git/index.lock` recovery

#### Failure mode

If a prior engine process crashed between `git add` and `git commit`, `.git/index.lock` is left on disk. Any subsequent git operation fails with `fatal: Unable to create .git/index.lock: File exists`. Today the `autoCommit` helper in `src/git/commit.ts` swallows this error and returns `null`, leaving the file on disk written but never committed — the repo drifts from git history silently.

#### Fix

Recovery runs in `MaadEngine.init()` during git layer initialization, before the engine is marked initialized. Sequence:

1. If `.git/index.lock` exists under `projectRoot`:
   - Check its mtime. If younger than 30 seconds, treat as a concurrent-process signal and refuse init with `GIT_ERROR` (`details: { reason: 'index-lock-recent', path, mtime }`). This protects against a second engine racing startup against a live one.
   - If older than 30 seconds, log a recovery action (`index_lock_stale_removed`) and delete the file. Append to `startupRecovery` array so `health()` and subsequent boot logs expose it.
2. After init, run `git status --porcelain` via the existing `GitLayer`. If dirty, log a `startup_git_dirty` warning and include the summary in `startupRecovery`. Do not auto-commit — the operator or the next write call cleans up organically.

The 30-second threshold matches journal-reconciliation practice already in `src/engine/journal.ts`. Not a tuning knob for 0.4.1.

#### Journal × git-lock interaction

If `journal.reconcile()` finds a pending `file_written` entry (write started but not committed) *and* the git lock is stale, recovery order is: clear the lock first, then the journal's file-written entry can be resolved on the next write attempt (the file is already on disk; journal just records the stage). No new logic needed in the journal.

### Error codes

Two new codes land with 0.4.1:

```ts
type ErrorCode =
  | ...existing...
  | 'WRITE_TIMEOUT'       // RESERVED for 0.8.5 — not emitted in 0.4.1
  | 'SHUTTING_DOWN';      // emitted by ops attempted during graceful shutdown
```

`WRITE_TIMEOUT` is reserved in the enum so clients can begin handling it without a breaking change when 0.8.5 activates the bounded-wait policy. 0.4.1 never emits it — the mutex blocks indefinitely.

### Test plan

All tests live in `test/engine/concurrency.test.ts`. Bun test runner with the existing harness. No external containers.

- **T1 — Two-writer race.** Two `createDocument` calls on distinct docs, started in parallel. Both succeed; both files on disk; backend shows both; git log shows exactly 2 commits.
- **T2 — N-writer flood.** 50 `createDocument` calls launched via `Promise.all`. All 50 succeed in FIFO order (verified via sequence-number docIds); no `.git/index.lock` lingers; peak `writeQueueDepth` observed > 1.
- **T3 — Writer + concurrent readers.** One writer + 20 parallel readers. Readers never wait measurably; read results are consistent (pre-write or post-write, not torn).
- **T4 — Stale git lock recovery.** Two cases: stale lock (mtime >30s old) is removed on init; recent lock refuses init with `index-lock-recent`.
- **T5 — Mutex releases on thrown error.** Force the write path to throw; subsequent call succeeds without hanging; queue depth returns to 0.
- **T6 — Bulk vs single write interleave.** Bulk of 20 + single write launched concurrently; bulk completes first, single commit comes after; no interleaving.

## Spec 2 of 5 — Idempotency Keys on Writes

Spec for optional client-supplied idempotency keys that deduplicate retried writes within a TTL window. Prevents the "retry on timeout creates a duplicate record" bug.

### Problem

With the 0.4.1 write mutex blocking indefinitely and network transports (0.5.0 HTTP/SSE arriving next) introducing new retry paths, a client that retries after a perceived timeout can land the same logical write twice. MAADB assigns a new docId on each request unless the caller specifies one, so two retries of a "create agent named Acme" produce two Acme records. Silent corruption of the database.

The existing `expectedVersion` mechanism on `updateDocument` protects against *concurrent* writes overwriting each other, but does nothing for *retries of the same write*.

### Goals

- Client can opt into exactly-once semantics for a write by supplying an idempotency key.
- Second occurrence of the same key within the TTL returns the *original* result, not a new one — including the same `docId`, `version`, and `validation` payload.
- No behavior change for clients that omit the key.
- Storage is bounded — old keys expire automatically.

### Non-goals

- **Automatic key generation on the server.** Key must be client-supplied; the server does not hash request bodies.
- **Cross-engine key sharing.** Keys are per-engine (per project).
- **Persistent key store.** Keys live in-memory. A restart clears them.

### Design

#### Where the check lives

At the **MCP tool layer**, not inside the engine. Reasoning:

- The engine's mutating methods already assume the caller's request is logically distinct. Pushing idempotency into `writes.ts` entangles the write pipeline with request-protocol concerns.
- MCP tool handlers are the natural boundary: they own the request envelope.
- If a future CLI or programmatic caller wants idempotency, they can layer their own replay cache.

Implementation location: `src/mcp/idempotency.ts` (new) plus thin hooks in each mutating tool handler (`src/mcp/tools/write.ts`).

#### Key shape

Client sends an `idempotencyKey` field in the tool arguments — a free-form string ≤128 chars, whatever UUID/hash scheme they prefer. Server treats it as an opaque identifier.

Scope: **per project, per tool**. Tuple `(projectName, toolName, idempotencyKey)` is the cache key. Same key across different tools is not a collision; keys are shared across all sessions bound to the same project.

**Why project-scoped, not session-scoped.** The main point of idempotency is retry survival. If a client's connection drops mid-retry and it reconnects with a fresh session ID, a session-scoped cache would miss and execute the write twice. Project-scoped cache replays cleanly regardless of session lifecycle. Key collision across unrelated clients is solved by clients using UUIDs — server does not police the key space (same contract as Stripe, AWS).

#### Cache

In-memory LRU with TTL, `src/mcp/idempotency.ts`:

```ts
interface IdempotencyEntry {
  key: string;                    // (sessionId, toolName, idempotencyKey) composed
  response: unknown;              // the original CallToolResult
  createdAt: number;              // ms epoch
}

class IdempotencyCache {
  private ttlMs = 10 * 60 * 1000;    // 10 minutes
  private maxEntries = 10_000;
  private entries = new Map<string, IdempotencyEntry>();

  get(key: string): IdempotencyEntry | null { /* + TTL check + LRU reorder */ }
  put(key: string, response: unknown): void { /* + evict oldest if over cap */ }
}
```

Single instance per MCP server process, shared across all sessions (the composed key includes `sessionId` so no cross-session leak).

TTL: **10 minutes**. Tunable via env `MAAD_IDEMPOTENCY_TTL_MS`.
Max entries: **10,000**. Simple LRU eviction. Tunable via env `MAAD_IDEMPOTENCY_MAX`.

#### Tool handler wiring

Every mutating tool (`maad_create`, `maad_update`, `maad_delete`, `maad_bulk_create`, `maad_bulk_update`) accepts an optional `idempotencyKey` in its input schema:

```ts
async function handle(args, extra) {
  const cacheKey = args.idempotencyKey
    ? composeKey(activeProject, 'maad_create', args.idempotencyKey)
    : null;

  if (cacheKey) {
    const cached = idempotencyCache.get(cacheKey);
    if (cached) {
      log('idempotency_replay', { tool, project: activeProject, key: args.idempotencyKey });
      return {
        ...cached.response,
        _meta: {
          request_id: currentRequestId,
          replayed: true,
          original_request_id: cached.originalRequestId,
          original_completed_at: cached.originalCompletedAt,
        },
      };
    }
  }

  const response = await /* normal write path */;

  if (cacheKey && responseWasSuccessful(response)) {
    idempotencyCache.put(cacheKey, {
      response,
      originalRequestId: currentRequestId,
      originalCompletedAt: new Date().toISOString(),
    });
  }
  return response;
}
```

Three deliberate choices:

1. **Only successful responses are cached.** A failed write should not be treated as "done" on retry.
2. **Replay is identifiable, not transparent.** Response body is byte-identical to the original. `_meta.replayed: true` plus `_meta.original_request_id` and `_meta.original_completed_at` let clients detect a replay if they care (telemetry, debugging). Clients that ignore `_meta` see the same response shape as a fresh call. This is a deliberate change from the earlier "fully transparent" design — identifiability costs nothing and makes post-mortems easier.
3. **No mutex interaction.** The cache check happens *before* acquiring the mutex. Replay hits skip the mutex entirely.

### Error code

```ts
| 'IDEMPOTENCY_REPLAY'   // emitted only in logs/audit channel, not in response
```

Not a visible error code to the client — the whole point is that replay is transparent. The code exists for log emission.

### Test plan

- **I1 — Same key returns cached response.** Two calls to `maad_create` with identical args + same key; second returns same docId/version/filePath; backend has one document.
- **I2 — Different keys produce different writes.** Two distinct keys; two distinct documents.
- **I3 — No key = no dedup.** Existing behavior preserved.
- **I4 — Failed write is not cached.** First call fails validation; second call (same key, valid fields) succeeds.
- **I5 — TTL expires.** Mocked clock advanced past 10 min; cache miss on replay.
- **I6 — Project scoping.** Same key across two sessions bound to the same project → replay. Same key across two sessions bound to different projects → two distinct writes.
- **I6b — Replay identifiability.** Replay response has `_meta.replayed === true` and `_meta.original_request_id` matching the original call's ops log line.
- **I7 — LRU eviction.** Fill cache to max, put one more, oldest evicted.

## Spec 3 of 5 — Per-Session Rate Limiting

Spec for a per-session token bucket that caps incoming request rate and payload size. Protects the engine from runaway agents and flood scenarios.

### Problem

An agent in a retry loop or a buggy automation can hammer the MCP server unbounded. Today there is no ceiling — the only back-pressure is whatever the transport imposes (none, for stdio). With HTTP/SSE and container-hosted deployments on the horizon, one bad client must not be able to saturate an engine's write mutex or exhaust process memory by sending huge payloads.

### Goals

- Bound request rate per MCP session, independent of transport.
- Bound single-request payload size to prevent memory exhaustion.
- Give callers a clean, machine-readable rejection when they exceed limits.
- Tunable via env.
- Zero cost on the happy path.

### Non-goals (deferred to 0.8.5)

- Per-token or per-tier rate limits.
- Global engine-wide rate limit.
- Queue-depth / backpressure distinct from rate-limit rejections.
- Burst credit accumulation across idle periods longer than 1 minute.

### Design

#### Token bucket, per session

| Bucket | Capacity (burst) | Refill rate | Applies to |
|---|---|---|---|
| `writes/sec` | 10 | 10 tokens/sec | any mutating tool call |
| `writes/min` | 60 | 60 tokens/60s | any mutating tool call (secondary gate) |
| `concurrent` | 5 | n/a (semaphore) | any tool call in flight |

Plus one per-request cap:

- `maxPayloadBytes`: 1,048,576 (1 MiB). Applies to the serialized tool-args JSON.

Read-only tool calls are exempt from write buckets but still count against `concurrent` and `maxPayloadBytes`.

#### Why two buckets for writes

The `writes/sec=10` bucket allows brief bursts. The `writes/min=60` bucket caps sustained rate. One cap alone can't express both "allow bursts" and "cap sustained."

#### Bucket storage

Per-session buckets in-memory in the MCP server process, keyed by `sessionId`. Garbage-collected on session disconnect.

```ts
class SessionRateLimiter {
  private writesSec = new Map<string, TokenBucket>();
  private writesMin = new Map<string, TokenBucket>();
  private concurrent = new Map<string, number>();

  tryAcquireWrite(sessionId: string): { ok: true } | { ok: false; reason: string; retryAfterMs: number };
  tryAcquireConcurrent(sessionId: string): { ok: true; release: () => void } | { ok: false };
  checkPayloadSize(bytes: number): { ok: true } | { ok: false; limit: number };
  disposeSession(sessionId: string): void;
}
```

Token refill is lazy — computed on-demand, not via background timer.

#### Env knobs

| Var | Default | Meaning |
|---|---|---|
| `MAAD_RL_WRITES_PER_SEC` | 10 | writes/sec bucket capacity + refill rate |
| `MAAD_RL_WRITES_PER_MIN` | 60 | writes/min bucket capacity + refill rate |
| `MAAD_RL_CONCURRENT` | 5 | max concurrent in-flight requests per session |
| `MAAD_RL_PAYLOAD_BYTES` | 1048576 | max serialized args size |
| `MAAD_RL_DISABLE` | unset | if `"1"`, bypasses all checks |

All read at startup; no hot-reload.

#### Rejection response

```json
{
  "ok": false,
  "errors": [{
    "code": "RATE_LIMITED",
    "message": "Rate limit exceeded",
    "details": {
      "reason": "writes_per_sec" | "writes_per_min" | "concurrent" | "payload_too_large",
      "limit": 10,
      "retryAfterMs": 450
    }
  }]
}
```

`retryAfterMs` is null for `concurrent` (retry when a slot frees) and `payload_too_large` (resending won't help).

#### Client retry contract

Clients observing `RATE_LIMITED` **MUST**:

1. Honor `retryAfterMs` when present.
2. Use exponential backoff on repeated rejection (double each time, cap at 60s).
3. Give up after ~5 consecutive rejections and surface to the human caller.
4. Treat `payload_too_large` as permanent — modify the request, don't retry.
5. Use idempotency keys (H4) on any write that might be retried. Without a key, a retry that lands after the original succeeded produces a duplicate.

The server does not enforce these rules — it is always allowed to reject. Clients ignoring the contract will be rate-limited more aggressively, not less.

#### Wiring

Rate limit check runs inside `withSession` / `withEngine`, after session/role validation and before the engine call. Canonical order (shared with all 0.4.1 items):

1. Session bound / role / project whitelist
2. Payload size check → `RATE_LIMITED`
3. Idempotency cache check → replay if hit
4. Concurrent cap check → `RATE_LIMITED`
5. Write rate check (if mutating) → `RATE_LIMITED`
6. Call engine method
7. Release concurrent slot in `finally`

Order rationale: payload check first because oversize args waste compute; idempotency before rate-limit because a replay should not consume tokens; concurrent release in `finally` so an engine error doesn't leak a slot.

### Test plan

- **R2 — Writes/min cap.** 60 creates at controlled rate; 61st fails; advance mocked clock 60s; succeeds.

## Spec 4 of 5 — Structured Logging + Audit Channel

Spec for structured JSON logging of every MCP tool call, plus a dedicated audit channel for write operations. Replaces the existing console-only text logger as the default output while preserving the internal `logger` primitive.

### Problem

Today `src/engine/logger.ts` emits plain-text lines to stderr via a pluggable `LogHandler`. That's enough for local development but insufficient for:

1. **Production debugging.** Grepping a text stream for what a single request did is error-prone. Structured JSON with a `request_id` lets an operator trace one request end-to-end.
2. **Compliance / audit.** A write-only channel with before/after state is what any operator eventually asks for. Mixing audit events with operational noise makes both harder to consume.
3. **Latency / error observability.** Without per-request `latency_ms` and `error_code`, operators can't spot slow tools or rising error rates.

### Goals

- Every MCP tool call produces exactly one structured log line, JSON, to stdout.
- Every successful write emits an additional line on a dedicated audit channel with before/after version.
- Sensitive values never appear in logs.
- Existing `logger` primitive continues to work — its handler is swapped.
- Log level controlled by env.

### Non-goals

- OpenTelemetry / distributed tracing. Deferred to 0.8.5.
- Metrics endpoint. Deferred to 0.8.5.
- Async log shipping (syslog, fluentbit). Pipe stdout.
- Rotation / archival. Downstream concern.

### Design

#### Library choice: pino

Adding `pino` as a production dependency:
- Fastest JSON logger in Node; near-zero overhead for disabled levels.
- Synchronous output option for tests and shutdown safety.
- First-class redaction via path spec.
- Widely understood by ops teams.

Alternative considered: hand-rolled JSON emitter. Rejected — reimplementing pino's redaction, level gating, and sync-mode support is a week of work for no upside.

#### Two loggers, one library

```ts
// src/logging.ts (new)
import pino from 'pino';

export const opsLog = pino({
  level: process.env.MAAD_LOG_LEVEL ?? 'info',
  redact: {
    paths: ['args.authorization', 'args.token', 'args.headers.authorization'],
    censor: '[redacted]',
  },
});

export const auditLog = pino({
  level: 'info',                  // always on
  base: { channel: 'audit' },
  // writes to MAAD_AUDIT_PATH if set, else stdout with channel tag
});
```

#### Ops log: one line per MCP call

Location: inside `withSession` / `withEngine` wrappers in `src/mcp/with-session.ts`:

```ts
opsLog.info({
  event: 'tool_call',
  request_id: requestId,
  session_id: sessionId,
  project: activeProject,
  tool: toolName,
  role: effectiveRole,
  payload_size: JSON.stringify(args).length,
  latency_ms: latencyMs,
  result: response.ok ? 'ok' : 'error',
  error_code: response.ok ? null : response.errors?.[0]?.code ?? null,
}, 'tool_call');
```

Every field on every call. Missing values are explicit `null` for stable parsing.

`request_id` also attached to response as `_meta.request_id` so clients can quote it in bug reports. Non-breaking (MCP allows extra fields).

#### Audit log: one line per successful write

Fires from MCP write-tool handlers after engine call succeeds:

Rule: **tool args are logged verbatim.** Future tools accepting sensitive values must add their field to the redact path list at registration time.

## Spec 5 of 5 — Per-Request Timeouts + Graceful Shutdown

Spec for bounded request execution and a clean SIGTERM path. Ensures container restarts never leave stale git locks, in-flight writes either complete or reject deterministically, and the engine exits in a known-good state.

### Problem

Today:

1. **No request timeout.** A tool call that hangs (slow filesystem, stuck git subprocess, pathological extraction) holds its slot indefinitely. Under rate-limit concurrent caps, a stuck request starves other callers.
2. **Abrupt shutdown.** `registerShutdownHooks` in `src/mcp/lifecycle.ts` calls `engine.close()` and `process.exit(0)`. In-flight writes can exit mid-operation, leaving files partially committed or `.git/index.lock` orphaned.
3. **No way to reject new work during shutdown.** New tool calls during the shutdown window race with exit.

Container orchestrators send SIGTERM and expect a graceful window (default 10s on Docker, 30s on K8s) before SIGKILL. MAADB must use that window deliberately.

### Goals

- Every MCP tool call has a bounded maximum duration.
- SIGTERM triggers a drain: new calls rejected, in-flight calls given bounded window to complete, process exits.
- Shutdown completes in-flight writes cleanly (mutex drains, journal settles, git commits land, SQLite closes, audit log flushes).
- No `.git/index.lock` ever left from a clean shutdown.
- Tunable grace window for operators with non-default orchestrator timeouts.

### Non-goals

- Active cancellation of in-flight writes. Drain waits, does not interrupt.
- Work queue persistence across restarts. Unstarted requests return `SHUTTING_DOWN`.
- Kubernetes preStop hooks in-engine. Operators wire externally.

### Design

#### Per-request timeout

Every tool call runs under a timeout ceiling, enforced in `withSession`:

```ts
const TIMEOUT_MS = Number(process.env.MAAD_REQUEST_TIMEOUT_MS ?? 30_000);

const response = await Promise.race([
  toolHandler(),
  new Promise<ToolResponse>((_, reject) => setTimeout(
    () => reject(new RequestTimeoutError(toolName, TIMEOUT_MS)),
    TIMEOUT_MS,
  )),
]);
```

On timeout:
- Response: `{ ok: false, errors: [{ code: 'REQUEST_TIMEOUT', ... }] }`.
- Behavior depends on *where* the request was — see Cooperative Cancellation below.
- Concurrent-slot release fires when the handler actually completes, not on the timeout.

Default 30s. Tunable via `MAAD_REQUEST_TIMEOUT_MS`.

#### Cooperative cancellation

Cancellation is *cooperative at stage boundaries*, not preemptive. An `AbortSignal` threads from the timeout into the handler; each stage boundary checks it before starting the next step. Once a stage is in motion, it completes.

| Request state at timeout | Action |
|---|---|
| Queued in mutex (not yet holding lock) | Remove from queue. Engine method never runs. Safe drop. |
| Holding mutex, between stages | Current stage completes. Signal checked at next boundary. `REQUEST_TIMEOUT` returned. Completed work stands; startup recovery handles partial journal entries on next boot. |
| Inside a single stage (mid git commit, SQLite transaction, fsync) | Stage runs to completion. `REQUEST_TIMEOUT` returned. Delayed `tool_call_overrun` log line. |

**What cooperative cancellation does NOT do:**
- It does not cancel an in-flight git commit, SQLite transaction, or `fsync`.
- It does not roll back a file that has been written to disk.
- It does not remove a backend index row that has been inserted.

The client receives `REQUEST_TIMEOUT`, but the server may still finish the write. Trade-off of "safe cancellation" vs "fast response." For write-heavy clients, a retry with an idempotency key (H4) replays the cached response rather than creating a duplicate. This is precisely why H4 exists.

#### Shutdown state machine

```
RUNNING  --SIGTERM-->  DRAINING  --drain complete or timeout-->  EXITING  -->  exit(0 or 1)
```

```ts
let state: 'running' | 'draining' | 'exiting' = 'running';
```

**RUNNING:** normal operation.

**DRAINING** — triggered by first SIGTERM or SIGINT:

1. Flip `state` to `'draining'`.
2. Emit ops log `event: 'shutdown_start'`.
3. Reject new tool calls. `withSession` entry check — if not `running`, return `SHUTTING_DOWN` immediately.
4. Wait for in-flight writes to finish. Poll at 50ms intervals until:
   - `writeLock.depth() === 0`
   - rate limiter's `totalInFlight() === 0`
5. If drain exceeds `MAAD_SHUTDOWN_TIMEOUT_MS` (default 10s), force-exit with `shutdown_timeout` warning.

**EXITING** — always runs, whether drain succeeded or timed out:

1. Close the engine (SQLite backend cleanly).
2. Flush the audit log.
3. Dispose per-session rate-limit state.
4. Emit ops log `shutdown_complete` with `elapsed_ms` and `drain_timed_out` boolean.
5. `process.exit(drainTimedOut ? 1 : 0)`.

Exit code 1 on timed-out drain surfaces the issue to operators monitoring container exit codes.

#### Second SIGTERM / SIGINT

If another signal arrives during `draining`: accelerate. Skip rest of drain wait, jump to EXITING with `shutdown_forced` log. Operators who mean it can override.

#### Mutex interaction

Drain waits for `writeLock.depth() === 0`. Since `SHUTTING_DOWN` rejects new calls at the wrapper level, nothing new enters during drain.

**Edge case:** a write currently blocked in the mutex queue when SIGTERM arrives. Two options:

- **Option A: Let queued writes execute.** They're already accepted; rejecting now creates an "is my write saved?" ambiguity that defeats idempotency keys. Drain waits for all of them.
- **Option B: Reject queued writes.** Fail fast.

**Choice: A.** If operators need faster drain they set a tighter `MAAD_SHUTDOWN_TIMEOUT_MS`.

#### Idempotency and rate-limit state

Both in-memory, discarded at exit. Documented:

- Post-restart idempotency retries run as fresh writes.
- Post-restart rate-limit buckets are refreshed.

Cross-restart idempotency is a larger storage question deferred to a future version.

#### Error codes

```ts
| 'REQUEST_TIMEOUT'    // per-request timeout exceeded
| 'SHUTTING_DOWN'      // server is draining or exiting
```

`SHUTTING_DOWN` reserved in the concurrency spec's error code section; this spec is the module that emits it.

#### Env knobs

| Var | Default | Meaning |
|---|---|---|
| `MAAD_REQUEST_TIMEOUT_MS` | 30000 | per-request timeout ceiling |
| `MAAD_SHUTDOWN_TIMEOUT_MS` | 10000 | drain grace window before forced exit |

### Test plan

- **S1 — Request timeout fires mid-stage.** Slow handler (35s) with `MAAD_REQUEST_TIMEOUT_MS=1000` → returns `REQUEST_TIMEOUT` in ~1.1s; delayed `tool_call_overrun` line on eventual completion; handler's side effects persist.
- **S1b — Request timeout fires while queued.** Mutex held by slow write; queued write times out → `REQUEST_TIMEOUT` without engine method running; backend has only the first doc.
- **S2 — Graceful drain completes.** Slow write in-flight + SIGTERM → write succeeds, process exits 0.
- **S3 — Drain timeout forces exit.** Slow handler + tight `MAAD_SHUTDOWN_TIMEOUT_MS=500` + SIGTERM → exit 1 in ~600ms; `shutdown_timeout` warning.
- **S4 — New requests during drain are rejected.** `maad_create` 100ms after SIGTERM → `SHUTTING_DOWN`.
- **S5 — Second SIGTERM accelerates.** Two SIGTERMs 100ms apart → immediate exit with `shutdown_forced`.
- **S6 — Queued writes execute on drain (Option A).** 3 writes queued + SIGTERM → all 3 complete.
- **S7 — Clean git state on exit.** Write + immediate SIGTERM → drain completes, no `.git/index.lock`.
- **S8 — Engine close is idempotent.** Explicit `engine.close()` + shutdown → no errors.

## Review questions

Pointed questions for reviewers. Each is a decision I'd like challenged before implementation starts:

1. **Mutex blocks indefinitely vs 30s timeout.** The deferred `WRITE_TIMEOUT` decision. Is "no timeout in 0.4.1" the right posture, or should there be a generous ceiling (say 60s) that prevents a truly-stuck process from holding everyone forever? My current read: no — if a write is stuck 60s, something is broken at a level no timeout will recover. Logs will make it visible.

2. **Idempotency cache TTL = 10 minutes.** Too short, too long, or roughly right? Real retry windows I've seen in practice are 30s–5min. 10 min covers that with margin; longer means more memory per entry without obvious benefit.

3. **Rate limit defaults (10 writes/sec, 60 writes/min, 5 concurrent, 1 MiB payload).** Starting points. Am I under- or over-estimating? Happy to be wrong — these become env-driven so operators can tune.

4. **Reads exempt from write rate buckets.** Rationale: reads are cheap (pointer lookups against SQLite), write bucket's purpose is protecting the mutex. Concurrent cap still applies. Is this the right tradeoff, or should there be a separate `reads/sec` bucket to protect against query-bomb attacks?

5. **Audit log: file separation via `MAAD_AUDIT_PATH`, not a dedicated transport.** Means operators who want centralized audit have to configure file+shipper. Should 0.4.1 go further and bake in a stream-to-syslog or similar? My read: no, stdout piping covers the 90% case and saves a dependency.

6. **Graceful shutdown Option A (queued writes drain) vs Option B (reject on SIGTERM).** Currently A. Reviewer preference?

7. **Per-request timeout default = 30s.** Bulk operations can exceed this. Operators will bump it for bulk-heavy workloads. Better default, or tool-specific timeouts?

## Resolved after first round of review

- **Idempotency scope changed from `(session, tool, key)` to `(project, tool, key)`.** Reviewer correctly pointed out that session-scoped cache misses the main use case: a client whose connection drops mid-retry reconnects with a fresh session ID and re-issues the write, creating a duplicate. Project-scoped cache replays cleanly across reconnects.
- **Idempotency replay is now identifiable, not transparent.** Response body is byte-identical to the original; `_meta.replayed: true` plus `_meta.original_request_id` and `_meta.original_completed_at` let clients detect a replay for telemetry/debugging without breaking the transparent-retry contract. Costs ~4 lines of code, high debuggability value.
- **Cooperative cancellation adopted at stage boundaries.** Reviewer pushed for full cancellation on timeout; that's unsafe for git commits / SQLite transactions / fsync which are not interruptible at the OS level. Middle ground: `AbortSignal` threaded into handlers, checked at stage boundaries. Requests that time out while still queued in the mutex are dropped cleanly; requests mid-execution complete and emit `tool_call_overrun`.
- **Rate limit client-contract language added.** Clients MUST honor `retryAfterMs`, use exponential backoff, cap retries at ~5 consecutive rejections, treat `payload_too_large` as permanent, and use idempotency keys on retries. The server doesn't enforce these rules but clients ignoring them will be rate-limited more aggressively.
- **Mutex `finally` release.** Already in the original spec via `try/finally` in `runExclusive`. Test T5 specifically verifies release-on-thrown-error. Confirmed, no change needed.
