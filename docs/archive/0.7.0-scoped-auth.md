## Problem

MAADB 0.5.0 shipped HTTP transport with a **single shared bearer** (`MAAD_AUTH_TOKEN`) and a server-wide `--role` flag. Any holder of the secret gets the same role across every project — no per-agent identity, no per-token role scoping, no per-project access control on a per-token basis. That's fine for a single-operator deployment; it breaks three things the multi-agent demo needs:

1. **Identity attribution.** Git commits and audit logs today record only a role, not *which agent* did the write. When three agents share one brain, "who created this record?" is unanswerable.
2. **Per-agent role differentiation.** The reader-agent and writer-agent and admin-agent all authenticate with the same shared secret, so the engine can't enforce different roles per agent.
3. **Per-project scoping.** An agent granted access to project A can silently reach project B through the same shared bearer.

Covers scenarios 5 (identity attribution) and 7 (per-agent roles) from the 8-scenario support matrix.

## Solution shape

A file-backed token registry with per-token identity, role, and per-project scoping. Every HTTP request resolves its bearer to a token record; the record's identity and capabilities propagate through the session context into the existing role-enforcement chokepoint at `with-session.ts:128`, into the audit log, and into git commit metadata.

Three load-bearing design rules:

1. **Three-cap effective role model.** Effective role at any tool call = `min(projectCeiling, tokenProjectCap, requestedRole)`. Every cap must be ≥ the tool's minimum or `INSUFFICIENT_ROLE`.
2. **Immutable token records.** Capability changes require `revoke` + `issue-token`. Three operations total: `issue`, `rotate`, `revoke`. Rotation changes the secret only, never the capabilities.
3. **HTTP-transport-scoped.** The registry validates bearers presented in the HTTP `Authorization` header. Stdio has no bearer channel — it retains the existing synthetic-mode semantics unchanged (`--role` flag, `SessionState.token = null`, no audit identity). Extending identity to stdio (e.g. via MCP initialize params) is a separate future concern.

## Token registry

File: `_auth/tokens.yaml` at instance root (not per-project — auth is engine-level; project scoping is handled via per-token allowlist).

### Record shape

```yaml
tokens:
  - id: tok_01HZ7QJ5K9X3V8R2N1M4          # stable handle for audit + management
    hash: sha256:5d41402abc4b2a76b9719d9...  # SHA-256 of plaintext bearer; plaintext NEVER stored
    role: writer                              # global role ceiling for this token
    agent_id: agt-claude-dev                  # identity pointer (nullable if user_id set)
    user_id: null                             # identity pointer (nullable if agent_id set)
    name: claude-laptop                       # display label
    projects:                                 # per-token project allowlist
      - name: brain
        role: writer                          # optional per-project role cap (≤ global role)
      - name: atlas
        role: reader
    expires_at: null                          # ISO datetime or null for no expiry
    created_at: 2026-04-20T15:00:00Z
    revoked_at: null                          # set on revoke; record retained for audit
```

### Required identity

At least one of `agent_id` / `user_id` MUST be set. Anonymous tokens are rejected at issuance — every audit entry must be attributable.

### Project allowlist semantics

**Shape: object entries only.** Every entry is `{ name, role? }`. No bare strings. `name: "*"` is a wildcard matching any project in `instance.yaml`. `role` is optional — defaults to the token's global role.

```yaml
projects:
  - name: "*"              # wildcard entry
    role: reader           # wildcard's per-project cap (optional)
  - name: admin-project
    role: admin            # explicit entry; overrides wildcard for this project
```

**Lookup rule:** **explicit names always override the wildcard, regardless of list order.** At pin time, the engine first looks for an exact-name match; only falls back to the wildcard entry if no exact match exists.

| Configuration | Pin attempt → effective project cap |
|---|---|
| `projects: []` (empty) | Token rejected at issuance — `TOKEN_IDENTITY_REQUIRED` companion: no project access is not a valid token |
| `projects: [{ name: "foo" }]` | `foo` → token's global role; any other project → `TOKEN_PROJECT_FORBIDDEN` |
| `projects: [{ name: "foo", role: reader }]` | `foo` → reader (capped below global role) |
| `projects: [{ name: "*" }]` | Every instance project → token's global role |
| `projects: [{ name: "*", role: reader }]` | Every instance project → reader |
| `projects: [{ name: "*", role: reader }, { name: "brain", role: admin }]` | `brain` → admin (explicit wins); every other project → reader |
| `projects: [{ name: "*" }, { name: "secrets", role: reader }]` | `secrets` → reader; every other project → global role |

Per-project `role` can only **reduce** the token's effective capabilities, never escalate above the global `role`. Validated at issuance (`TOKEN_ROLE_ABOVE_GLOBAL`).

**Newly added projects (via instance hot-reload):** wildcard-tokens pick up new projects automatically on next pin attempt. Explicit-entry tokens don't — they need `revoke + reissue` to gain access to projects added after issuance.

### Token plaintext format

`maad_pat_<32hex>` — prefix + 32 hex chars (128 bits random). Prefix makes tokens recognizable in configs, logs, and leaked-credential scanners. Hex is chosen for copy/paste safety across shell contexts.

The plaintext is returned **once** by `issue-token` / `rotate-token` and never persisted. Only the SHA-256 hash lives in `tokens.yaml`.

## Three-cap effective role (load-bearing)

Every tool call resolves its effective role through three caps:

```
effectiveRole(project, token, requested) =
    min(
        instance.projects[project].role,     # CAP 1: project ceiling (instance.yaml)
        token.projectCaps.get(project),      # CAP 2: token's per-project cap (tokens.yaml)
        requested                            # CAP 3: client's requested role at bind time
    )
```

All three must be ≥ the tool's minimum role (`getMinRoleForTool(toolName)`) or the call fails `INSUFFICIENT_ROLE`.

### Why three caps

- **Cap 1 (project ceiling)** protects a project from any client exceeding the declared role — current semantic, unchanged.
- **Cap 2 (token cap)** is the new layer — the same shared bearer no longer grants the same role to everyone holding it.
- **Cap 3 (requested role)** preserves voluntary downgrade, e.g. a writer-capable agent binding as reader for a read-only session. Existing `as=<role>` semantics unchanged.

### Compatibility with 0.4.0 project-role semantics

Cap 1 still comes from `instance.yaml`. If a token claims `role: admin` on a project declared `role: writer` in the instance config, the project ceiling wins — token sees writer. No privilege escalation vector from the registry.

## No-mutability rule (load-bearing)

**Token records are immutable except for the three operations below.** Any capability change (role, identity, project allowlist) requires `revoke-token` + `issue-token`. This is a design decision, not an oversight — see `dec-maadb-069-scoped-auth` rationale.

| Operation | Changes | Preserves |
|---|---|---|
| `issue-token` | Everything — new id, hash, metadata | — |
| `rotate-token` | `hash` only (new plaintext secret) | `id`, `role`, `identity`, `projects`, `expires_at`, `created_at` |
| `revoke-token` | Sets `revoked_at` | Everything else (record retained for audit trail) |

Rationale for immutability:

- **Audit clarity.** "This token has role writer" is a fact with a creation date. Making role mutable means every audit query needs a time-range join against a mutation log.
- **Least surprise for token holders.** An agent holding a token should not silently gain `admin` because someone ran a CLI command. The token is a capability certificate; escalation deserves a new certificate.
- **Blast-radius reasoning.** If a token leaks and you are debugging, knowing `role` and `identity` are immutable lets you compute exactly what the attacker could do from issuance — a fully-mutable model forces correlation of mutation timelines.

## Rotation semantics

`rotate-token` changes the secret without changing capabilities. Use cases: scheduled key hygiene, suspected leak, proactive cycling.

```bash
maad auth rotate-token tok_01HZ7QJ5...
```

- New plaintext generated, new SHA-256 hash replaces `hash`
- `id`, `role`, `identity`, `projects`, `expires_at`, `created_at` unchanged
- Old plaintext stops working immediately (no grace period in 0.6.9 v1)
- Returns new plaintext once; never shown again

Rotation refuses on revoked tokens (`TOKEN_REVOKED`) or expired tokens (`TOKEN_EXPIRED`) — issue new instead.

Grace-period rotation (`--grace-period 5m` for zero-downtime rewiring) is **deferred to 0.7.0** unless operator friction surfaces it sooner.

### No OAuth-style refresh tokens

MAADB's auth is internal-service auth — the token lives in a config file on a trusted machine (stdio or HTTPS transport). Adding a refresh-token protocol solves problems MAADB does not have (untrusted browser clients, third-party consent, user-level session split). The bearer-with-expiry-and-rotation model is correct for this trust boundary.

Client-layer auth (OAuth, SSO, user login, consent) is the **application layer's concern** (brain-app or any gateway you build in front). Brain-app speaks OAuth to end users and presents MAADB with a single shared bearer. The engine stays clean.

## In-flight request semantics

Token mutations land without restart. Tool calls in flight when a mutation happens:

| Event mid-call | Behavior |
|---|---|
| Token revoked | In-flight completes. Next request wire-fails with a plain 401; ops log records `TOKEN_REVOKED` |
| Token rotated | In-flight completes. Next request with new plaintext succeeds; old plaintext's hash no longer resolves in registry → plain 401, ops log records `TOKEN_UNKNOWN` |
| Token expiry hit | In-flight completes. Next request wire-fails with a plain 401; ops log records `TOKEN_EXPIRED` |
| tokens.yaml hot-reload (SIGHUP) | In-flight completes on old snapshot; next request uses new snapshot |

No tool handler aborts mid-execution due to auth state changes. Auth is checked once at tool-call entry — matches existing `with-session.ts` flow.

## Hot-reload mechanism

Three triggers reload `tokens.yaml` into running memory:

| Trigger | Source | Behavior |
|---|---|---|
| CLI mutation (`issue-token`, `revoke-token`, `rotate-token`) | Local operator | Writes YAML atomically + signals engine in-process (same-process) or via file-watcher (cross-process) |
| `SIGHUP` | Operator / systemd reload | Re-reads `_auth/tokens.yaml` + `instance.yaml` — shares path with fup-2026-058 if shipped |
| `maad_instance_reload` MCP tool | Admin agent (requires admin role) | Admin-only; accepts `{reloadInstance: true, reloadTokens: true}` flags |

The first two ship in 0.6.9. The MCP tool depends on fup-2026-058 and ships when that lands.

If `fup-058` ships first, 0.6.9 piggybacks on its SIGHUP path. If not, 0.6.9 implements a standalone SIGHUP handler for `tokens.yaml` only, and refactors into the shared path when fup-058 arrives.

## Identity propagation

### Session context

`SessionState` gains a `token` field set at transport-auth time (before `maad_use_project(s)` binds the session):

```typescript
interface TokenContext {
  tokenId: string;                     // tok_01HZ...
  role: Role;                          // global role from registry
  projectCaps: Map<string, Role>;      // per-project caps
  identity: {
    agentId?: string;                  // agt-claude-dev
    userId?: string;
    name?: string;
  };
}

// In SessionState (extends 0.4.0 shape)
interface SessionState {
  // ... existing fields
  token: TokenContext | null;          // null in stdio synthetic mode
}
```

### Git commit messages

When `MAAD_COMMIT_IDENTITY=true` (new env flag), commit messages gain an identity prefix:

```
BEFORE (0.6.8):
  maad: create jrn-2026-072

AFTER (0.6.9 with flag):
  [role:writer agent:agt-claude-dev] create jrn-2026-072
```

Flag defaults **off** in 0.6.9 v1 until fup-2026-066 (bulk-write git commit bug) resolves. Identity-enriched commits amplify blast radius of any missed commits, so gate behind explicit opt-in.

### Audit log entries

Every audit event (`tool_call`, `session_open`, `session_close`, `auth_failure`, `auth_event`) gains identity fields:

```json
{
  "ts": "2026-04-20T15:30:00Z",
  "event": "tool_call",
  "session_id": "...",
  "tool": "maad_create",
  "project": "brain",
  "role": "writer",
  "token_id": "tok_01HZ7QJ5...",
  "agent_id": "agt-claude-dev",
  "user_id": null,
  "name": "claude-laptop",
  "result": "ok"
}
```

`auth_failure` events (401/403 paths) emit with `token_id: null` when no token resolved — no identity leak in failure logs.

### maad_health.sessions breakdown

`maad_health` response gains `sessions.byProject` (role distribution scoped per project) and `sessions.byIdentity` (per-agent/user session count) aggregates:

```json
{
  "ok": true,
  "data": {
    "sessions": {
      "total": 7,
      "byProject": {
        "brain":  { "reader": 2, "writer": 4, "admin": 1 },
        "atlas":  { "reader": 1, "writer": 2 }
      },
      "byIdentity": {
        "agt-claude-dev": 3,
        "agt-codex-cli": 2,
        "agt-master-dev": 1,
        "unattributed": 1
      }
    }
  }
}
```

**Aggregation semantics:**
- `total` counts distinct sessions (one session = one MCP connection, regardless of how many projects it's bound to).
- `byProject[project][role]` counts `(session, project)` pairs: a session bound to `brain` as writer and `atlas` as reader contributes +1 to `byProject.brain.writer` and +1 to `byProject.atlas.reader`. The inner sums therefore exceed `total` when multi-project sessions exist — this is intentional; the shape answers "how many sessions have writer role on brain right now?" directly.
- `byIdentity` counts distinct sessions per identity (does not multi-count across projects).
- `unattributed` counts sessions in single-bearer legacy mode (no token registry entry).

A flat `byRole` aggregate is intentionally omitted — since a single session carries per-project roles, there's no unambiguous session→role mapping to flatten on.

## Error codes

New codes introduced in 0.6.9 — split into **wire-visible** (surface in MCP response `error.code` or 403 response bodies) and **ops-log-only** (never exposed on the wire; audit trail only).

### Wire-visible codes

| Code | When | Wire surface |
|---|---|---|
| `TOKEN_PROJECT_FORBIDDEN` | Token's `projects` allowlist rejects the pinned project | 403, MCP `error.code` |
| `TOKENS_FILE_INVALID` | `tokens.yaml` parse error at boot or live reload | Startup fatal / `INSTANCE_RELOAD_FAILED` on live reload |
| `TOKEN_IDENTITY_REQUIRED` | `issue-token` called without `--agent-id` or `--user-id` | CLI error; also `maad_issue_token` MCP error |
| `TOKEN_PROJECT_UNKNOWN` | `issue-token --project foo` where `foo` not in `instance.yaml` (or, with `--verify`, agent record not in project) | CLI / MCP tool error |
| `TOKEN_ROLE_ABOVE_GLOBAL` | Per-project role cap > token's global role | CLI / MCP tool error at issuance |

### Ops-log-only codes (401 path)

All four cases emit a plain **401 Unauthorized** response with **no body detail** — the 0.5.0 wire shape is preserved byte-for-byte. The specific reason lands in the structured ops log under `auth_failure` events with the corresponding `reason` code.

| Code | When | Wire surface |
|---|---|---|
| `TOKEN_UNKNOWN` | Bearer doesn't match any registered hash | 401, empty body; ops log only |
| `TOKEN_REVOKED` | Hash matches but `revoked_at` is set | 401, empty body; ops log only |
| `TOKEN_EXPIRED` | Hash matches but `expires_at` is past | 401, empty body; ops log only |
| `TOKEN_MALFORMED` | Authorization header present but malformed (no `Bearer ` scheme, empty token) | 401, empty body; ops log only |

**Rationale:** Wire responses never tell the caller *why* auth failed. That keeps the unauth response shape byte-for-byte identical to 0.5.0 (T27) and avoids leaking whether a presented token was once-valid-now-revoked vs never-valid. Operators see the distinction in the ops log; attackers see 401.

  --role reader|writer|admin \
  --agent-id <agt-docId>            # at least one identity flag required

## Backward compatibility

### HTTP boot-mode semantics — legacy single-bearer hard-removed

**BREAKING in 0.7.0** (per `dec-maadb-071-legacy-bearer-removal`): the 0.5.0 legacy single-bearer mode is removed. `_auth/tokens.yaml` is required for HTTP mode. No fallback path.

| `_auth/tokens.yaml` state | `MAAD_AUTH_TOKEN` | HTTP boot | Outcome |
|---|---|---|---|
| Absent | Set | ❌ refuses — `LEGACY_BEARER_REMOVED` | Boot error includes migration hint: "Generate tokens.yaml via `maad auth issue-token`" |
| Absent | Unset | ❌ refuses — `TOKENS_FILE_MISSING` | Boot error includes the same hint |
| Present, parse error | Any | ❌ refuses — `TOKENS_FILE_INVALID` with line number | — |
| Present, empty `tokens:` array | Any | ❌ refuses — `TOKENS_FILE_EMPTY` | Registry must have ≥1 active entry to serve HTTP |
| Present, ≥1 valid entry | Unset | ✅ starts | **Registry mode** (primary path) |
| Present, ≥1 valid entry | Set | ✅ starts | **Registry mode**; `MAAD_AUTH_TOKEN` value is looked up in the registry like any other bearer — either it matches a registry hash (works) or it doesn't (`TOKEN_UNKNOWN` on every request) |

**Why the hard removal:** MAADB 0.6.x had zero external HTTP consumers. The one internal HTTP deployment (brain-app on the droplet) is operator-owned and migrates in ~5 minutes. Preserving legacy mode forever carried dead fallback code, mixed-mode semantics in docs, and a second error-path tree for a consumer that doesn't exist. Removal simplifies `resolveToken()` by one branch, drops the mixed-mode section from operator docs, and makes the auth surface easier to reason about.

### Migration from 0.6.x

One-time procedure on each HTTP deployment:

1. **Issue a gateway token** — from build machine or the droplet itself:
   ```bash
   maad auth issue-token --role=admin --name=<deployment-name> --projects='*'
   # emits maad_pat_<32hex> ONCE to stdout
   ```
   This also creates `_auth/tokens.yaml` if it didn't exist and writes the hashed entry.

2. **Update the deployment's env** — swap the old `MAAD_AUTH_TOKEN` value for the new `maad_pat_<32hex>` token. The env key stays `MAAD_AUTH_TOKEN` — it's still a bearer, just now validated against the registry instead of a shared string comparison.

3. **Restart** — `sudo systemctl restart maadb.service` (or equivalent).

4. **Verify** — `maad_health` in an admin session shows `sessions.byIdentity` populated; audit log entries carry `token_id` + identity fields.

Total time: ~5 minutes. The only MAADB HTTP deployment in the fleet today follows this recipe once.

### Synthetic / stdio mode

Unchanged. `SessionState.token` is `null`; `effectiveRoles` comes from `--role` as today. The token registry is HTTP-transport-only — stdio has no bearer channel to validate against.

### `maad_instance_reload` coexistence

When `tokens.yaml` is absent, HTTP mode didn't boot — the tool is unreachable by definition. When present, reload loads the registry alongside `instance.yaml`.

## Implementation phases

Four phases. Roughly 3.5–4 engine-days focused.

### Soft

## Non-goals

Explicitly out of scope. Named so they don't scope-creep the PR.

- **External IdP integration** (OIDC / JWT / OAuth). Application layer's concern — brain-app handles it at the Next.js edge. Engine stays bearer-only.
- **Refresh token protocol.** OAuth-style access+refresh solves problems MAADB does not have. Covered in rotation + expiry.
- **Token mutability.** Zero mutable fields. Capability changes require revoke + reissue.
- **Grace-period rotation** (`--grace-period`). Deferred to 0.7.0 unless operator friction surfaces sooner.
- **Per-tool rate limiting by token tier.** Deferred to 0.9.0 Remote MCP Hardening.
- **Token quotas** (max tokens per identity / per instance). Deferred.
- **IP pinning / geo restrictions.** Operational hardening, not an engine-level concern.
- **Agent record provisioning.** `--agent-id` is a pointer; CLI `--verify` flag cross-checks existence but does not create. Agent record creation stays on the self-registration / brain-app-server-side paths.
- **User / group membership.** `user_id` is a free-form slug today. Group membership, ACL inheritance, RBAC hierarchies are post-1.0.
- **Commit-identity default-on.** Stays opt-in until fup-066 resolves.

## Risk / rollout

- **Additive by construction.** Deployments without `tokens.yaml` see zero behavior change. 575+ existing tests pass unchanged.
- **Legacy single-bearer mode stays the default.** Operators opt into scoped auth by creating `tokens.yaml`. No forced migration.
- **Three-cap composition is a single function.** Every effective-role bug is localized to one testable point.
- **401 response shape preserved byte-for-byte.** T27 locks this — no identity leak through unauth responses.
- **SIGHUP fallback path is minimal.** If fup-058 takes longer, 0.6.9 standalone handler is ~20 lines.
- **Estimated effort: 3.5–4 engine-days focused.** Phase 1 (primitives) ~0.5 day; Phase 2 (middleware + three-cap) ~1 day; Phase 3 (identity + CLI + MCP tools) ~1.5 days; Phase 4 (reload + docs) ~0.5 day.

## Ship gate

Not a brain-app launch blocker — brain-app uses the single shared bearer at the Next.js edge. Multi-agent demos need this to ship cleanly. Recommended ship window: between 0.6.8 (current) and 0.7.0 (npm prep), ideally before any external multi-agent demo.

## Decision record

See `dec-maadb-069-scoped-auth-identity-design-lo` in the brain for the locked design decisions:

1. Version slot: 0.6.9 (point bump, not 0.7.0 breaking)
2. Location: instance root, not per-project
3. Token format: `maad_pat_<32hex>` prefix
4. Per-token project allowlist with optional per-project role override
5. Backward compat: single-bearer fallback preserved
6. Role tiers: reader / writer / admin (unchanged from 0.4.0)
7. Three-cap effective role model
8. No mutability — three operations only (issue / rotate / revoke)
9. No OAuth refresh — bearer + expiry + rotation
10. Grace-period rotation deferred to 0.7.0
11. `MAAD_COMMIT_IDENTITY` gated off until fup-066 resolves
